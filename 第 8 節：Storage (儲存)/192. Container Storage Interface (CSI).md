# 192. Container Storage Interface (CSI)

## 1. 🏷️ 課程定位

- **影片標題：** 192. Container Storage Interface (CSI)

## 2. 📌 核心概念摘要
CSI (Container Storage Interface) 是一套跨平台的標準化 API (RPC) 規範。它的核心目標是將「儲存驅動程式」與「Kubernetes 核心代碼」徹底解耦 (Out-of-Tree)。透過 CSI，各大儲存供應商 (如 AWS EBS、Dell EMC) 只需要遵守這套標準寫一支外掛程式，就能讓 Kubernetes 自動化地幫你建立、掛載與刪除遠端雲端硬碟。

## 4. 🔑 知識點擷取 (Detailed Notes)
這張簡報的精華在於「職責分離」，以下是您必須掌握的底層邏輯：

- **誕生背景 (In-Tree 走向 Out-of-Tree)：**
  - **過去 (In-Tree)：** 早期所有儲存廠商的程式碼都被硬寫進 Kubernetes 的原始碼裡面。這導致 K8s 程式碼極度肥大，且如果 AWS 要更新一個 EBS 的 bug，必須等 K8s 整體發布新版本。
  - **現在 (Out-of-Tree / CSI)：** K8s 不再內建這些驅動，而是只提供「CSI 介面」。廠商自己維護自己的 CSI 外掛 (通常以 Pod 形式部署在叢集中)。

- **RPC (遠端程序呼叫) 規範：**
  - 如截圖中央所示，CSI 定義了一系列標準動作。
  - **編排器端 (K8s) 的責任：** 當開發者建立 PVC (PersistentVolumeClaim) 時，K8s 負責「呼叫 (`SHOULD call`)」對應的 RPC (如 `CreateVolume`)。
  - **儲存端 (Vendor) 的責任：** CSI 驅動程式接收到訊號後，必須負責去底層實體設備上「真正劃分出空間 (`SHOULD provision`)」，並回報給 K8s。

- **跨平台通用性：** 
  CSI 不是 K8s 專屬的。只要符合 CSI 標準，同一套 AWS EBS 的儲存驅動，也可以直接拿給 Mesos 或 Cloud Foundry 使用。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在 CKA 考場中，我們不會去寫 CSI，但我們必須知道如何查詢叢集中安裝了哪些 CSI 驅動：

```bash
# 💡 查詢叢集中目前註冊了哪些 CSI 驅動程式 (CSIDriver)
kubectl get csidrivers

# 💡 查看 StorageClass 底層使用了哪個 CSI Provisioner (配置器)
# 注意 Provisioner 欄位，通常長得像 ebs.csi.aws.com
kubectl get storageclass

# 💡 考場除錯神技：當 Pod 一直掛載不到 Volume 時，可以檢查 VolumeAttachment 物件
# 這是 K8s 用來追蹤 CSI 磁碟是否已經成功掛載到特定 Node 上的內部物件
kubectl get volumeattachment
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
### 🎯 考試情境預測：
考題可能會要求你建立一個 StorageClass，並明確要求它的 provisioner 必須設定為某個 CSI 驅動（例如：`provisioner: csi-hostpath`）。接著你要建立一個 PVC 去觸發這個 CSI 幫你產生 PV (這稱為動態配置 Dynamic Provisioning)。

### 🛑 避坑指南 (Volume 卡死陷阱)：
在雲端環境 (如 AWS) 使用 CSI 時，一個 EBS Volume 一次只能掛載到一台 Node (因為 `ReadWriteOnce` 限制)。如果 Node-A 突然死機 (NotReady)，CSI 無法正常收到 `DeleteVolume` 的脫離訊號，這塊硬碟就會一直被死鎖在 Node-A 上。導致 Pod 在 Node-B 重啟時永遠卡在 `ContainerCreating`。

### 🔧 Troubleshooting：
- **現象：PVC 狀態一直卡在 Pending。**
  - **排查步驟：**
    1. `kubectl describe pvc <pvc-name>` 檢查 Events。
    2. 如果出現類似 `failed to provision volume with StorageClass ...` 的錯誤。
    3. 去檢查該 CSI 驅動的 Pod 是否正常運作！CSI 驅動通常安裝在 `kube-system` 命名空間下（會包含一個 Controller Deployment 和一個 Node DaemonSet）。去查閱這些 CSI Pod 的 logs，通常是因為 API 憑證錯誤或配額不足導致它們無法呼叫外部雲端的 API。

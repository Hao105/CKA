# 混血部署架構 (Advanced Architecture: Helm Post-Renderer)

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 13 節：(2025 Updates) Kustomize Basics (架構師實戰延伸篇)
- **影片標題**：272-1. Advanced Architecture: Helm Post-Renderer (混血部署模式的優缺點與版本規範)

## 2. 📌 核心概念摘要
「Helm 渲染骨架，Kustomize 注入靈魂」是目前應對複雜第三方應用的終極解決方案。其底層運作目標在於利用 Helm 的 `--post-renderer` 攔截點，在 YAML 送交 Kubernetes 叢集前的最後一刻進行「補丁加工」。

這就像是在工廠出貨前的**「流水線最終加工攔截站 (Assembly Line Interception Point)」**：標準化產品（Helm 渲染出的骨架）在出廠裝箱前被強行攔截，由本地專家（Kustomize）加上專屬客製化標籤與微調後再放行。這使得我們既能不碰原始碼強行修改任何屬性，又 100% 完整保留了 Helm 原生的生命週期管理 (Release, History, Rollback) 能力。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **嚴格版本要求 (Version Requirements)**：
  - Helm 必須在 **v3.1.0** 或以上版本。Helm 2 以及早期的 Helm 3.0 都不支援 `--post-renderer` 這個參數。目前的 Kubernetes 生態中（通常搭配 Helm v3.10+），此功能已極度穩定。
- **架構優點 (Pros)**：
  - **完整保留生命週期 (Lifecycle Management)**：部署出來的資源依然會被註冊為一個 Helm Release。你可以使用 `helm list` 看到它，更新壞了可以直接 `helm rollback` 瞬間救回，這是純 Kustomize （如上節所述 `apply -k` 結合 Helm 的做法）做不到的。
  - **免 Fork 零污染 (Zero-Forking)**：面對官方 Chart (如 Nginx Ingress) `values.yaml` 不支援的詭異客製化需求，不需要將整個 Chart 下載回來自己維護魔改，完美保持與官方上游版本的同步升級能力。
- **架構缺點與限制 (Cons & Limitations)**：
  - **黑箱效應與除錯困難**：當部署報錯時，第一時間很難分辨是「Helm 預設的 values 給錯了」還是「Kustomize 的 Patch 打錯位置了」。除錯路徑必須拆成兩段進行。
  - **CI/CD 環境依賴加重**：執行部署的 Runner 或本地環境，必須**同時安裝** `helm` 與 `kustomize` 兩個執行檔，而且作業系統必須支援 Bash (用來跑 wrapper 腳本)。若在極簡的 Alpine 容器內跑管線，可能會遇到找不到指令的窘境。
  - **補丁脫靶風險**：如果 Helm 上游的官方人員在改版時，修改了某個 Deployment 的 name 或標籤結構，你的 Kustomize Patch 就會因為找不到目標而「脫靶 (Target not found)」導致部署崩潰。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在實務中，我們必須先準備好一個攔截腳本（例如 `kustomize-wrapper.sh`）：

```bash
#!/bin/bash
# kustomize-wrapper.sh
# 注意：腳本必須有執行權限 (chmod +x kustomize-wrapper.sh)

# 1. 將 Helm 吐出的 YAML (從 stdin) 寫入本地的暫存檔
cat <&0 > base.yaml

# 2. 呼叫 Kustomize 進行最終渲染 (stdout 將自動傳回給 Helm 接管)
kustomize build .
```

接著，使用以下指令進行測試與部署：

```bash
# 🔍 實務除錯心法：不要急著 install，先用 template 跑一次空轉渲染
# 這能在不污染叢集的情況下，將 Helm + Kustomize 結合的最終 YAML 印在螢幕上檢查
helm template my-release bitnami/redis --post-renderer ./kustomize-wrapper.sh

# 🚀 實戰部署：確認 YAML 無誤後，正式安裝並紀錄 Release
helm install my-release bitnami/redis --post-renderer ./kustomize-wrapper.sh

# 🚑 驗證生命週期：這時候如果 Kustomize 的補丁搞壞了服務，您可以直接依賴 Helm 退版！
helm rollback my-release 1
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> - **考場真相**：CKA 的考試範圍不會考到 `--post-renderer` 這麼進階的結合技。考題會將 Helm 和 Kustomize 拆成兩道獨立的題目來考。
> - **面試利器**：但如果您在考過 CKA 後去面試「資深雲端架構師 / DevOps 工程師」，面試官極高機率會問：「*如果官方 Helm Chart 不滿足需求（例如缺乏某個 Annotation 欄位），你會怎麼辦？*」流利回答這個混血方案，保證讓面試官眼睛一亮。

> [!WARNING]
> **🛑 避坑指南**：
> - **權限遭拒 (Permission Denied)**：執行 Helm 指令時，最常遇到的報錯是 `fork/exec ./kustomize-wrapper.sh: permission denied`。請立刻執行 `chmod +x ./kustomize-wrapper.sh` 給予腳本執行權限。
> - **資料流斷裂**：腳本內的 `cat <&0` 代表接收來自 Helm 的標準輸入。如果您自作聰明修改腳本，忘記將最後的處理結果導出到 `stdout` (如直接少了 `kustomize build` 輸出)，Helm 就會收到空文件而報錯。

> [!CAUTION]
> **🔧 Troubleshooting**：
> **升級時報錯 `patching failed` 或資源找不到**：
> - **降維排查法**：先拿掉 `--post-renderer` 參數，單獨執行 `helm template > test-base.yaml`。
> - 將 `test-base.yaml` 打開檢查裡面的 `kind` 和 `name`，通常是因為新版 Helm Chart 改變了資源名稱，導致您的 `kustomization.yaml` 裡面的 patches 找不到舊名字了。修正 Patch 的目標 (Target) 名稱即可解決。

## 7. 📝 YAML 骨架 (YAML Skeleton)
為了配合 `kustomize-wrapper.sh`，該目錄下必須準備一個相對應的 `kustomization.yaml`，這也是架構的最後一塊拼圖：

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# 將腳本產生的 base.yaml 視為基礎資源載入 (此時裡面充滿了 Helm 的骨架)
resources:
  - base.yaml

# 對被攔截的 Helm 資源進行強行補丁修改 (注入靈魂)
patches:
  - target:
      kind: Deployment
      name: my-release-redis  # 這裡的名稱必須精準命中 Helm 產生的名字
    patch: |-
      - op: add
        path: /spec/template/spec/securityContext
        value:
          runAsUser: 1000
```

## 8. 🧠 自我測驗
<details>
<summary>在混血部署模式下，如果我手動執行 <code>kubectl delete deployment my-release-redis</code>，然後再執行 <code>helm rollback my-release 1</code>，Deployment 會恢復並且依然帶有 Kustomize 注入的 Patch 嗎？</summary>

**解答：**
**會的！完美恢復並帶有 Patch。**
這就是 `--post-renderer` 的終極價值。因為最終經過 Kustomize 加工後的「完美 YAML」，已經透過 Helm 的機制被打包成 Release 紀錄，並加密儲存在 Namespace 的 Secret 中。
當你執行 `helm rollback` 時，Helm 是直接從它自己的 Secret 中抽出那個被保存的最終快照來還原，根本不需要重新執行渲染腳本，因此所有的客製化設定都會原封不動地恢復。
</details>

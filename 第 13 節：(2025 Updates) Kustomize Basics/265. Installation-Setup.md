# Kustomize 安裝與環境設定 (Installation/Setup)

## 1. 🏷️ 課程定位

- **影片標題**：265. Installation/Setup

## 2. 📌 核心概念摘要
本節的底層運作目標在於「建置獨立版的 Kustomize 執行環境 (Standalone CLI)」。

雖然 Kubernetes 原生 `kubectl` 已整合了基礎的 Kustomize 功能，這就像是**「手機內建的相機」**，能應付考場上的基本需求；但在實務 CI/CD 流程中，為了追求更複雜的渲染支援（例如結合 Helm）與外掛機制，我們必須手動安裝獨立版的 `kustomize`，如同換上**「專業單眼相機」**，徹底解鎖最新版本的進階功能。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **獨立版安裝機制 (Standalone Installation)**：
  - **定義**：將官方發布的 `kustomize` 二進位執行檔 (Binary) 下載，並手動配置到系統的環境變數路徑 (如 `/usr/local/bin`) 中。
  - **觸發條件**：當 `kubectl` 內建的 Kustomize 版本過舊，導致執行較新的 `apiVersion`（如 `v1beta1` 轉為 `v1`）或進階 Transformers 失敗時，就必須改用此獨立版。
- **版本驗證 (Version Check)**：
  - **定義與底層意義**：執行 `kustomize version --short` 可以快速確認二進位檔是否具有正確的執行權限並已被系統識別。
  - **重要性**：在除錯「版本相容性 (Version Skew)」問題時，確認當下使用的引擎版本（例如 `{kustomize/v4.4.1}`）是架構師的第一直覺動作。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
雖然考試不建議浪費時間安裝獨立版，但若在實務或特定考題中環境已提供該 CLI，請熟悉以下指令：

```bash
# 🔍 驗證獨立版 Kustomize 是否安裝成功與當前版本 (影片截圖重點)
kustomize version --short

# 🚀 實務操作：使用獨立版 CLI 將當前目錄的設定「渲染」成標準 YAML
# 💡 注意：這個指令只會輸出 YAML 到終端機，不會真的部署到 K8s 叢集
kustomize build .

# 🎯 實務組合技：使用獨立版編譯，並透過管線 (Pipeline) 傳遞給 kubectl 進行部署
kustomize build . | kubectl apply -f -
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> - **考場真相**：CKA 考試絕對不會考 `kustomize` 的下載與安裝設定！考場的終端機環境中，K8s 相關的工具都已就緒。
> - **解題策略**：遇到 Kustomize 的題目，請毫不猶豫地使用 `kubectl kustomize <目錄>` 或 `kubectl apply -k <目錄>` 即可拿分，別去管獨立版。

> [!WARNING]
> **🛑 避坑指南**：
> **PATH 變數未生效**：在實務上手動下載二進位檔後，如果終端機回報 `kustomize: command not found`，通常是因為你沒有給予執行權限（忘記下 `chmod +x kustomize`），或是沒有將它移動到系統的 `$PATH` 目錄下（例如 `/usr/local/bin`）。

> [!CAUTION]
> **🔧 Troubleshooting**：
> **本地端與叢集行為不一致**：當你在本地端用 `kustomize build .` 跑得好好的，但丟進 CI/CD 用 `kubectl apply -k .` 卻報錯時。
> **除錯動作**：請立刻使用 `kustomize version` 與 `kubectl version --client` 比對兩者的 Kustomize 引擎版本。這幾乎 100% 是因為你的獨立版寫了太新的語法，但 K8s 內建版太舊看不懂所導致的版本衝突。

## 7. 📝 YAML 骨架 (YAML Skeleton)
為何我們有時會遇到版本衝突？看看下方的 `apiVersion` 差異，這就是獨立版與內建版最常見的陷阱：

```yaml
# 舊版 Kustomize (舊版 kubectl 內建) 通常支援：
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - deployment.yaml

---
# 較新獨立版 kustomize (v4+) 可能支援或強制要求更穩定的版本：
# 🛑 陷阱：若在舊版 kubectl 執行含以下 apiVersion 的 YAML，將會觸發無法識別的報錯！
apiVersion: kustomize.config.k8s.io/v1
kind: Kustomization
resources:
  - deployment.yaml
```

## 8. 🧠 自我測驗
<details>
<summary>我的 CI/CD Pipeline 腳本中寫了 <code>kustomize build . | kubectl apply -f -</code>。這句指令的後半段 <code>-f -</code> 代表什麼意思？</summary>

**解答：**
`kubectl apply -f -` 中的最後一個減號 (`-`) 是一個 Linux 標準約定，代表**「從標準輸入 (Standard Input, STDIN) 讀取資料」**。
也就是說，它會接收前一個指令 `kustomize build .` 輸出在螢幕上的純 YAML 文本，直接作為要部署的目標清單，而不需要先存成暫存檔案 (`temp.yaml`) 再去讀取它。這是 Kubernetes 實務上極為優雅且常見的組合技。
</details>

# 核心控制檔解析 (The kustomization.yaml File)

## 1. 🏷️ 課程定位

## 2. 📌 核心概念摘要
在 Kustomize 的架構中，`kustomization.yaml` 是整個宣告式管理的**總指揮樞紐 (Entrypoint)**。

這就像是一份**「頂級廚師的食譜」**：它清楚列出了需要哪些「原始食材」(`resources` 基礎清單)，以及要套用哪些「獨家調味與烹飪手法」(`customizations` 轉換規則)。它的底層目標是奉行「渲染與部署分離」的原則，讓基礎架構的修改可以在本地端完全預覽與驗證後，再安全地送入 Kubernetes 叢集。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **核心控制檔 (`kustomization.yaml`) 的組成**：
  - **定義**：Kustomize 引擎在執行時，會自動在指定目錄下尋找此檔案，將其視為進入點。
  - **內容結構**：它主要包含兩大區塊：
    1. **資源清單 (Resources)**：明確列出 Kustomize 應該管理並合併的基礎 Kubernetes Manifests (YAML 檔)。
    2. **客製化規則 (Customizations)**：定義要套用在這些資源上的轉換邏輯（例如：統一加上特定的 Label、替換 Image 版本、或是注入 ConfigMap）。
- **編譯指令 (`kustomize build`) 的底層行為**：
  - **觸發機制**：執行 `kustomize build <目錄>` 時，引擎會讀取 `kustomization.yaml`，將資源與規則進行「記憶體內合併 (In-memory merge)」與「轉換」。
  - **限制條件 (Limitations)**：它絕對不會與 K8s API Server 溝通！這個指令純粹是一個「本地端的文字處理器」，它不會將任何資源部署到叢集上。
- **部署輸出的重導向 (Redirection)**：
  - **底層對象變化**：因為 `build` 指令只會將最終渲染的 YAML 輸出到終端機螢幕上 (stdout)，實務上必須將這個輸出結果透過管線 (Pipeline) 重導向給 `kubectl apply` 指令，才能真正完成部署。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在實務與考場中，我們必須熟練切換「純渲染」與「直接部署」的指令：

```bash
# 🔍 考場/實務驗證：只做本地渲染，預覽最終 YAML 結果（不部署）
# 💡 這等同於獨立版的 `kustomize build .`
kubectl kustomize ./my-kustomize-dir

# 🚀 考場必備捷徑：跳過 pipe 重導向，直接使用原生 kubectl 將渲染結果部署進叢集
# 💡 這是 CKA 考場中最快且唯一推薦的做法
kubectl apply -k ./my-kustomize-dir

# 🛠️ 實務管線做法：使用獨立版 CLI 編譯，並透過管線 ( | ) 交給 kubectl 部署
# 注意最後面的 `-`，代表讓 kubectl 從標準輸入 (stdin) 讀取檔案
kustomize build ./my-kustomize-dir | kubectl apply -f -

# 📝 實務除錯/稽核：將渲染好的完整 YAML 導出為實體檔案，方便後續檢查或版控
kustomize build ./my-kustomize-dir > final-manifest.yaml
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> 題目會要求您建立或修改一個 `kustomization.yaml`，並將結果部署。
> **配分重點**：考官的系統腳本只會檢查資源是否正確建立在指定的 Namespace 內。因此，您不需要糾結要用 `kustomize build` 還是 `kubectl kustomize`，請一律使用 `kubectl apply -k` 快速拿分。

> [!WARNING]
> **🛑 避坑指南**：
> - **檔名嚴格限制**：Kustomize 引擎只認得三個特定檔名：`kustomization.yaml`, `kustomization.yml`, 或大寫的 `Kustomization`。若您拼字錯誤（例如少打一個 o），執行 `kubectl apply -k` 時會直接報錯找不到設定檔。
> - **資源檔案不存在**：在 `kustomization.yaml` 的 `resources:` 列表下宣告的 YAML 檔案，必須真實存在於相對目錄中。如果有列出但檔案找不到，渲染過程會立刻中斷並報錯。

> [!CAUTION]
> **🔧 Troubleshooting**：
> **合併驗證失敗**：當您執行 `kubectl apply -k` 出現 Schema Validation Error（驗證錯誤）時，這代表您的 Patch 寫錯了，導致合併出來的 YAML 結構不合法。
> **架構師除錯法**：立刻退一步，執行 `kubectl kustomize . > debug.yaml`。打開 `debug.yaml`，您會非常清楚地看到到底是哪一個區塊的縮排壞掉或打錯字。

## 7. 📝 YAML 骨架 (YAML Skeleton)
一份標準的 `kustomization.yaml` 食譜通常長這樣：

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# 1. 原始食材 (基礎資源清單)
resources:
  - deployment.yaml
  - service.yaml

# 2. 調味手法 (客製化規則)
# 為所有資源統一加上前綴與標籤
namePrefix: dev-
commonLabels:
  app: my-web-app
  env: development
```

## 8. 🧠 自我測驗
<details>
<summary>如果我執行 <code>kustomize build .</code> 跑出了一大串完美的 YAML 內容在螢幕上，這時候 Kubernetes 叢集裡面已經建立好這些資源了嗎？</summary>

**解答：**
**還沒！**
`kustomize build` 純粹是一個「本地端的文字處理器與渲染引擎」，它與 K8s API Server 之間沒有任何連線與溝通。除非你將它的輸出結果交給 `kubectl apply -f -`，或是直接使用內建整合的 `kubectl apply -k .`，否則你的叢集內不會發生任何改變。
</details>

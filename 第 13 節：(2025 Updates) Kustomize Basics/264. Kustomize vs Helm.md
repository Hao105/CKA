# Kustomize vs Helm：結合實務架構探討 (Kustomize vs Helm)

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 13 節：(2025 Updates) Kustomize Basics
- **影片標題**：264. Kustomize vs Helm (結合實務架構探討：結合兩者優點)

## 2. 📌 核心概念摘要
Helm 的強項在於套件管理與強大的範本邏輯，但代價是使用了非標準 YAML 的 Go Template 導致可讀性差；而 Kustomize 勝在純原生 YAML 的無痛疊加 (Patching)。

在實務架構中，結合兩者的核心心法就像是**「建造房子」**：「讓 Helm 擔任**營造廠 (Base Generator)**，快速搭建複雜的鋼筋水泥骨架；讓 Kustomize 負責最後的**室內裝潢 (Last-Mile Patcher)**，針對不同環境微調風格。」這讓我們既能享受開源社群的強大模板，又能強行無痛覆蓋客製化設定，達到完美互補。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **Helm 的不可取代性 (痛點反轉)**：
  - **優勢**：提供強大的 Conditionals (條件判斷)、Loops (迴圈) 與 Hooks (生命週期掛鉤)。
  - **角色定位**：用來對付結構極度複雜的第三方服務（例如 Prometheus Stack、GitLab）。我們不用自己從頭寫 YAML，直接借用開源社群的心血。
- **Kustomize 的不可取代性 (痛點解藥)**：
  - **優勢**：Helm 的模板經常因為混雜了 Go 語法而變得「難以閱讀 (hard to read)」且「非合法 YAML」。Kustomize 透過純 YAML 打補丁的方式，解決了這個痛點。
  - **角色定位**：當 Helm 的預設 `values.yaml` 沒有開放我們想改的欄位（例如：為特定 Pod 加上公司專屬的 Security Context 或 Annotation）時，Kustomize 就能強行無痛覆蓋，而不必去 Fork 整個 Helm Chart。
- **結合策略：宣告式整合 (Kustomize 內建 Helm 支援)**：
  - 直接在 `kustomization.yaml` 中宣告你要拉取哪個 Helm Chart。Kustomize 引擎在執行時，會自動在背景呼叫 Helm 將其展開為純 YAML，接著再套用你的 Patch。這也是 GitOps（如 ArgoCD, Flux）中最推薦的標準做法。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
雖然 CKA 考試中通常會將這兩個工具分開考，但作為架構師，必須掌握這種高階宣告式寫法。

```bash
# 🎯 考場/實務通用：加上 --enable-helm 參數，讓 Kustomize 知道要處理 Helm 區塊
# 在不部署的情況下，預覽最終結合 Helm 渲染與 Kustomize 覆蓋的完美 YAML 結果
kubectl kustomize ./my-dir --enable-helm

# 🚀 直接將結合後的結果部署到 K8s 叢集
kubectl apply -k ./my-dir --enable-helm
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> 考題極高機率會將兩者拆開：一題要求你用 `helm upgrade` 修改參數；另一題要求你用 `kubectl apply -k` 部署不同環境的 Overlays。
> **配分重點**：清楚認知到 Helm 產生的資源會帶有 Helm 專屬的 Annotations 與 Labels（被視為一個 Release，由 Helm 管理），而純粹用 Kustomize 部署的資源則依賴 K8s 原生的 Garbage Collection。

> [!WARNING]
> **🛑 避坑指南**：
> **版本相容性**：如果你在 `kustomization.yaml` 中使用 `helmCharts`，必須確保執行環境的 Kustomize 版本夠新（Kustomize v4.1.0+），且環境中**必須已經安裝了 helm 的執行檔**，否則 Kustomize 無法在背景呼叫它，將會直接報錯。

> [!CAUTION]
> **🔧 Troubleshooting**：
> **合併失敗或 Patch 沒打上去**：如果在結合使用時，發現你寫的 Patch 沒有生效，請先執行 `kubectl kustomize ./ --enable-helm > debug.yaml`。打開 `debug.yaml` 檢查，通常是因為 Kustomize Patch 中指定的 `name` 或 `kind` 與 Helm 渲染出來的資源名稱不一致，導致「補丁打錯位置」。

## 7. 📝 YAML 骨架 (YAML Skeleton)
假設我們要部署 Bitnami 的 Redis，但想透過 Kustomize 強行加上標籤，以下是結合兩者的 `kustomization.yaml` 範例：

```yaml
# 檔案：kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# 1. 宣告 Helm Chart (將 Helm 當作 Base 骨架)
helmCharts:
  - name: redis
    repo: https://charts.bitnami.com/bitnami
    releaseName: my-redis
    version: 18.0.1
    # 可以在這裡傳遞基礎的 Helm values
    valuesInline:
      architecture: standalone

# 2. 宣告 Kustomize Patch (對 Helm 展開後的結果進行最後一哩路的覆蓋)
commonLabels:
  environment: production
  managed-by: kustomize-helm-combo
```

## 8. 🧠 自我測驗
<details>
<summary>當我使用 <code>kubectl apply -k . --enable-helm</code> 部署了包含 Helm Chart 的 Kustomize 配置後，如果我執行 <code>helm list</code>，會看到這個 Release 嗎？為什麼？</summary>

**解答：**
**不會看到**。
因為 `kubectl apply -k` 只是利用 Helm 引擎在「本地環境」中將 Chart 渲染為純 YAML 文本，然後透過原生的 `kubectl apply` 提交給 Kubernetes API Server。過程中並沒有真正的 Helm Release 被安裝（不會在 K8s 中建立紀錄 Helm 狀態的 Secret），因此 `helm list` 無法追蹤到它。這些物件已經完全轉化為由原生 K8s 直接管理的資源。
</details>

# 資源渲染輸出與精準清理 (Kustomize Output & Deletion)

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 13 節：(2025 Updates) Kustomize Basics
- **影片標題**：267. Kustomize Output (結合截圖重點：Delete with Kustomize 資源清理操作)

## 2. 📌 核心概念摘要
Kustomize 不僅負責「渲染與部署」，它同樣掌管著資源的「生命週期終結」。本節的底層運作目標在於展示如何安全且精準地清理 (Delete) 由 Kustomize 建立的 K8s 資源。

這就像是**「憑收據退回整份客製化套餐」**：因為 Kustomize 在部署時可能會動態修改資源名稱或標籤（客製化餐點），我們不能單憑記憶去逐一退還單品（手動刪除）。我們必須將原始的 `kustomization.yaml` (收據) 遞交給引擎，讓它「反向解析」出完整的最終清單，再交由 API Server 進行一次性、無殘留的精準刪除。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **Kustomize 資源清理的底層邏輯**：
  - **定義**：當使用 Kustomize 部署包含多個 YAML 檔（如 Deployment + Service + ConfigMap）的應用時，不應該使用傳統的 `kubectl delete deployment <name>` 來逐一刪除，這會導致架構管理碎片化與資源殘留。
  - **觸發機制**：必須將整個 Kustomize 目錄作為目標傳入，讓系統一次性讀取 `kustomization.yaml` 並反向清理所有關聯對象。
- **名稱動態變更的陷阱 (Limitations)**：
  - **重要性**：在進階實務中，`kustomization.yaml` 經常會使用 `namePrefix` (前綴) 或 `nameSuffix` (後綴) 來動態修改資源名稱。這表示叢集內實際運行的名稱，與你本地端撰寫的 Base YAML 名稱完全不同。
  - **解決方案**：直接使用 `-k` 參數觸發刪除，引擎會自動在本地計算出那些被「加工過」的最終名稱，保證能精準命中目標，避免報錯「找不到該資源 (NotFound)」。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
從截圖畫面中可以看到兩種標準的刪除流派。在考場上，請務必熟練第一種：

```bash
# 🚀 考場必備捷徑：使用原生 kubectl 一次性刪除該目錄下定義的所有資源
# 這是最乾淨、最安全的做法，保證不會殘留孤兒資源
kubectl delete -k k8s/

# 🛠️ 實務管線做法：先用獨立版 CLI 編譯，並透過管線 ( | ) 交給 kubectl 刪除
# 💡 注意最後面的 `-`，這代表讓 kubectl 從標準輸入 (stdin) 讀取前一個指令產生的 YAML 清單
kustomize build k8s/ | kubectl delete -f -
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> 題目可能會要求你「清理掉 Namespace `web-prod` 中由 `/opt/kustomize/prod` 目錄所部署的所有資源，以釋放 Node 空間」。
> **解法**：不需要去查裡面到底部署了什麼，直接指定路徑執行 `kubectl delete -k /opt/kustomize/prod -n web-prod` 即可快速拿分。

> [!WARNING]
> **🛑 避坑指南**：
> - **目錄路徑錯誤**：執行刪除指令時，路徑必須指向包含 `kustomization.yaml` 的資料夾。如果指令下錯層級，會直接報錯 `error: path does not contain a kustomization.yaml`。
> - **避免手動干預**：如果在執行 `delete -k` 之前，你已經手動刪除了其中的某個 Service，執行刪除指令時可能會跳出 `NotFound` 警告。雖然這不影響其他資源的後續刪除，但在考場上最好保持操作的一致性。

> [!CAUTION]
> **🔧 Troubleshooting**：
> **刪除卡住 (Terminating)**：如果執行 `kubectl delete -k` 後終端機一直卡住無法結束，通常是因為其中包含的資源（如 PersistentVolumeClaim 或 Namespace）被底層的 Finalizers 阻擋。
> **除錯動作**：按下 `Ctrl+C` 中斷等待。使用 `kubectl get all` 找出狀態卡在 Terminating 的資源，接著使用 `kubectl edit <resource> <name>` 刪除 YAML 內的 `finalizers` 區塊，即可強制完成清理。

## 7. 📝 YAML 骨架 (YAML Skeleton)
為何我們絕對需要用 `-k` 來刪除？請看以下 `kustomization.yaml` 的前綴設定陷阱：

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - deployment.yaml # 裡面原本寫 metadata.name: web-app

# Kustomize 轉換規則
namePrefix: prod-
nameSuffix: -v2
```
*💡 架構師解析：雖然你的基礎 YAML 檔寫的是 `web-app`，但 Kustomize 實際部署到叢集裡的名字是 `prod-web-app-v2`。如果你手動執行 `kubectl delete deployment web-app`，K8s 絕對會回報找不到資源。使用 `kubectl delete -k .` 才能讓引擎幫你反向推導出正確的目標名稱。*

## 8. 🧠 自我測驗
<details>
<summary>如果我修改了 <code>kustomization.yaml</code>，把裡面的 <code>namePrefix</code> 從 <code>dev-</code> 改成 <code>prod-</code>，然後直接執行 <code>kubectl delete -k .</code>，原本以 <code>dev-</code> 開頭的舊資源會被刪除嗎？</summary>

**解答：**
**不會被刪除！**
這是一個實務上非常容易犯的致命錯誤。`kubectl delete -k .` 是根據「當下目錄內」的設定來推導要刪除的名稱。當你改成 `prod-` 後才執行刪除，引擎會去找名為 `prod-xxx` 的資源來刪，導致原本的 `dev-xxx` 變成永遠殘留在叢集裡的「孤兒資源 (Orphaned Resources)」。
**正確做法**：必須先用舊的設定檔執行 `kubectl delete -k` 清理乾淨後，再修改設定檔中的 Prefix 並重新部署。
</details>

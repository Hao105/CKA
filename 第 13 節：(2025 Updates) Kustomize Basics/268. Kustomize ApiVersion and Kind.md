# Kustomize 專屬 API 與 Kind 解析 (ApiVersion & Kind)

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 13 節：(2025 Updates) Kustomize Basics
- **影片標題**：268. Kustomize ApiVersion & Kind

## 2. 📌 核心概念摘要
`kustomization.yaml` 不是一個會被直接部署到 Kubernetes 叢集內部運行的實體資源（如 Pod 或 Deployment），而是專門寫給「Kustomize 渲染引擎」看的設定檔。

這就像是一組**「本地端專屬的通關密語 (Secret Handshake)」**：它必須宣告專屬的 `apiVersion` 與 `kind`，讓本地的 Kustomize 引擎能立刻識別並啟動，依據規則將底層的基礎 YAML 檔案合併與渲染後，才把最終標準的 Kubernetes 資源送進 API Server。API Server 本身其實根本看不懂這份特殊的設定檔！

## 4. 🔑 知識點擷取 (Detailed Notes)
- **專屬的 `apiVersion` 與 `kind`**：
  - **定義**：必須嚴格寫為 `apiVersion: kustomize.config.k8s.io/v1beta1` (或更新的 `v1`) 以及 `kind: Kustomization`。
  - **底層意義**：當你執行 `kubectl apply -k` 時，K8s API Server 本身並不懂這是什麼。是 `kubectl` 客戶端在本地端先透過這個特殊的 `apiVersion` 啟動 Kustomize 引擎，處理完畢後，才把轉換出的**標準** K8s 資源往叢集送。
- **`resources` 區塊 (資源清單)**：
  - **定義**：用來指定哪些「基礎 Kubernetes YAML 檔案」要被 Kustomize 引擎納管與讀取。
  - **限制條件 (Limitations)**：檔案必須與 `kustomization.yaml` 位於同一個目錄或相對子目錄下。如果檔案名稱打錯或不存在，引擎在渲染階段就會直接報錯中斷。
- **`commonLabels` 等轉換器 (Transformers)**：
  - **觸發行為**：例如設定了 `company: KodeKloud`。引擎會將這個 Label **全域注入**到 `resources` 清單中所有的 K8s 資源裡。不僅 Deployment 本身會加上這個 Label，連 Deployment 底下的 PodTemplate 以及 Service 的 selector 都會被智慧地同步修改，確保服務關聯性不斷鏈。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在 CKA 考場中，手寫 `kustomization.yaml` 是很常見的考題，請務必記住其基礎骨架，或者知道如何快速生成：

```bash
# 🎯 考場必備：手動快速建立標準的 kustomization.yaml 骨架
# 如果考題要求你為現有的 yaml 檔建立 kustomize 管理
cat <<EOF > kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - my-deployment.yaml
  - my-service.yaml
EOF

# 🚀 考場驗證：寫完設定檔後，務必先用 kustomize 指令「預覽」渲染結果
# 確保 apiVersion 沒拼錯，且基礎資源都有被正確載入
kubectl kustomize .

# ⚡ 如果預覽畫面沒有報錯，就可以放心地一鍵部署
kubectl apply -k .
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> 考官可能會給你一個已經寫好的 `kustomization.yaml`，但裡面故意把 `apiVersion` 拼錯，或是 `kind` 寫成了小寫的 `kustomization`，導致你無法部署。
> **解題**：使用 `vim` 打開該檔案，修正為 `kustomize.config.k8s.io/v1beta1` 與**大寫開頭**的 `Kustomization` 即可修復並成功 `apply`。

> [!WARNING]
> **🛑 避坑指南**：
> - **大小寫極度敏感**：YAML 內的 `kind: Kustomization` 的 **K** 必須大寫；但在終端機執行的指令是 `kubectl apply -k` 或 `kubectl kustomize` (**全小寫**)；設定檔名也是 `kustomization.yaml` (**全小寫**)。考場上若精神不濟很容易在這裡打轉。
> - **Version Skew (版本錯位)**：實務上，若你在本地用最新的獨立版 Kustomize，`apiVersion` 可能會寫 `v1`；但若 CI/CD 伺服器上的 `kubectl` 內建版本太舊，它可能只認得 `v1beta1`。這是在架構設計時必須統一工具鏈版本的原因。

> [!CAUTION]
> **🔧 Troubleshooting**：
> - **錯誤訊息**：`error: kustomization.yaml is not a valid kustomization file` 或 `unknown apiVersion/kind`。
> - **排解動作**：這 100% 是因為第一行的 API 或第二行的 Kind 拼錯了。
>   **可以下 `kubectl explain` 查嗎？** **絕對不行！** 因為這是 Kustomize 本地引擎專屬的格式，叢集端的 API Server 根本不認得它，`explain` 指令不會理你。請直接參考 Kubernetes 官方文件，或將這個固定的開頭格式死記下來。

## 7. 📝 YAML 骨架 (YAML Skeleton)
請牢記這個無法透過 `kubectl explain` 查詢的核心骨架：

```yaml
# 這是寫給「本地引擎」看的專屬設定檔
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - deployment.yaml
  - service.yaml

commonLabels:
  company: KodeKloud
```

## 8. 🧠 自我測驗
<details>
<summary>如果我在考場忘記了 Kustomization 的 <code>apiVersion</code> 怎麼拼，我可以輸入 <code>kubectl explain kustomization</code> 來查詢嗎？為什麼？</summary>

**解答：**
**不行！你會收到無法找到資源的錯誤。**
因為 `kubectl explain` 是向 Kubernetes API Server 查詢叢集內「已知支援的實體資源結構（如 Pod, Ingress 等）」。而 `kustomization.yaml` 僅供**客戶端本地的 Kustomize 引擎**閱讀，API Server 根本不知道這份文件的存在，自然也無法為你解釋它的結構。
**解決方案**：你必須使用 k8s 官方文件的搜尋功能，或是平時就將 `kustomize.config.k8s.io/v1beta1` 熟記於心。
</details>

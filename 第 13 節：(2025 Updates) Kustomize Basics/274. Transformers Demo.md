## 2. 📌 核心概念摘要
- **多重轉換疊加效應**：Kustomize 允許在單一 `kustomization.yaml` 中宣告多重轉換規則（如 Namespace、Labels、Annotations、Images）。執行渲染時，它會一次性地將所有規則套用到宣告的資源中，實現全域且動態的覆寫。
- **多環境標準化神器**：這證明了 Kustomize 能夠在完全不更動底層（Base）YAML 的情況下，重塑 Kubernetes 資源，是實現多環境（Dev/QA/Prod）組態管理的終極武器。
- **💡 生動比喻**：就像是一條「自動化汽車組裝線」。Base YAML 是原廠的「標準白車身」；而 `kustomization.yaml` 就是「客製化改裝車間」。車身經過這條生產線（Kustomize Build）時，會被同時噴上特定顏色的烤漆（Namespace 統一）、貼上車隊專屬貼紙（Labels/Annotations 附加），甚至連引擎都被悄悄換成不同的型號（Image 替換），最終產出完全客製化的車輛，但原廠的設計圖（Base）從未被修改。

## 4. 💻 CKA 必備實作指令
> **考場神技：在考場中面對這類綜合題型，先預覽再部署是鐵律！**

```bash
# 1. 👁️ 綜合渲染預覽 (Dry-run)
# 當寫了一堆 Transformer 後，務必先用這行檢查有沒有拼錯、縮排錯誤或漏掉資源
kubectl kustomize ./

# 2. 🚀 正式套用所有轉換並部署
# 系統會將設定的 namespace、新 labels 與替換後的 image 一併送給 API Server
kubectl apply -k ./

# 3. 🔍 驗證特定 Namespace 下的資源狀態
# ⚠️ 由於使用了 namespace transformer，檢查時千萬別忘了加上 -n 參數！
kubectl get all -n debugging
```

## 5. 🛠️ 實戰與最佳實踐
> [!IMPORTANT]  
> **考場情境預測**：CKA 考題極可能會提供一個包含多個 YAML 檔的資料夾，要求您建立 `kustomization.yaml` 並同時達成三大目標：
> 1. 將所有資源部署到特定 Namespace（如 `cka-staging`）。
> 2. 加上共通標籤（如 `env=staging`）。
> 3. 將特定 Deployment 的 Image 升級到指定版本。

> [!WARNING]  
> **地雷區：拼字與層級縮排**：
> - 拼寫錯誤：`commonLabels` 常被誤拼成 `commonLabel` (少 s)，或 `images` 拼成 `image`。這在考場上**不會跳出明顯錯誤**，只會默默導致該規則失效！
> - 縮排錯誤：所有 Transformer（`namespace`, `commonLabels`, `images` 等）必須與 `resources` 處於**同一層級**（最左側頂格對齊），絕對不能縮排進去。

> [!TIP]  
> **Troubleshooting 降維排錯 SOP**
> - **問題：** 下達 `kubectl apply -k .` 後，資源跑進了 `default` namespace，沒有進到預期的 `debugging`？
> - **排查步驟：**
>   1. 立刻執行 `cat kustomization.yaml`。
>   2. 檢查是否忘記寫 `namespace: debugging`，或拼成了錯誤的單字如 `namespaces`。
>   3. 使用 `kubectl kustomize .` 預覽，直接檢視輸出的 `metadata.namespace` 欄位，是最快確認語法有無成功生效的方式。

## 6. 📄 YAML 骨架
最標準的綜合型 `kustomization.yaml` 寫法範例：

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# 1. 載入原始資源
resources:
  - api-depl.yaml
  - db-depl.yaml

# 2. 全域 Metadata 轉換器 (必須最左側頂格，與 resources 同層)
namespace: debugging

commonLabels:
  env: staging
  department: backend

commonAnnotations:
  logging: verbose

# 3. 映像檔轉換器
images:
  - name: mongo         # Base YAML 中的原始名稱
    newName: postgres   # 欲替換的新名稱
    newTag: "13.4"      # 欲替換的新標籤
```

## 7. 🧠 自我測驗
<details>
<summary>題目 1：在 kustomization.yaml 中加入 <code>commonLabels: {env: staging}</code>，除了修改 Deployment 的 Metadata 之外，還會產生什麼「連動」效應？</summary>
<b>解答：</b>它會自動連動修改 Deployment 內的 <code>spec.selector.matchLabels</code> 以及 Pod Template 內的標籤 (<code>spec.template.metadata.labels</code>)。更強大的是，如果 Base 中包含 Service 資源，Service 的 <code>spec.selector</code> 也會被一併自動修改，確保網路路由不會因為標籤更動而斷線。
</details>

<details>
<summary>題目 2：為什麼我寫了 <code>commonLabel: {app: web}</code>，執行渲染後資源卻沒有加上這個標籤，而且 kubectl kustomize 也沒有報錯？</summary>
<b>解答：</b>因為發生了「拼字錯誤」。正確的參數名稱是 <code>commonLabels</code>（字尾有 <b>s</b>）。Kustomize 遇到不認識的頂層欄位名稱時，通常會默默略過而不主動報錯，這是考場上排錯時最容易忽略的地雷。
</details>

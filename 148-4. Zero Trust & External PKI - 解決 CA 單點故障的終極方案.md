## 2. 📌 核心概念摘要
導入「中繼憑證（Intermediate CA）」的目的不是為了拯救 K8s 叢集，而是為了「控制爆炸半徑（Blast Radius）」以拯救整個企業；而在最嚴苛的金融級環境中，為了解決 K8s 內部 `ca.key` 被竊的必死局，架構師會選擇「把造物主的印章徹底請出 K8s 叢集」，交由外部的硬體金庫（如 HashiCorp Vault）來代為蓋章。

## 3. 📊 流程圖與視覺化重現 (ASCII / Mermaid)
以下是頂級企業（如銀行、大型科技廠）真實運作的「外部 CA 與跨系統信任」架構：

```mermaid
graph TD
    subgraph Enterprise [企業核心內網 / 實體金庫]
        RootCA[企業總部 Root CA <br> 絕對離線死鎖] -->|簽發中繼憑證| Vault[HashiCorp Vault / 外部憑證中心 <br> 擁有強大防駭機制]
    end

    subgraph Legacy [外部傳統系統]
        VM[傳統資料庫 VM] -.->|信任企業 Root CA| RootCA
    end

    subgraph K8s [Kubernetes 叢集]
        API[API Server]
        Client[新加入的 Worker Node] -->|1. 遞交 CSR 申請表| API
        API -->|2. K8s 自己不蓋章! <br> 轉發給外部金庫| Vault
        Vault -->|3. 嚴格審核後蓋章回傳| API
        API -->|4. 核發合法 CERT| Client
        
        Client -.->|5. 跨系統連線 (mTLS)| VM
    end
```

## 4. 🔑 知識點擷取 (Detailed Notes)
針對你的兩個質疑，業界的真實解答如下：

**迷思破解 1：中繼憑證無法跨越 K8s 以外的系統？**
- **其實可以！這正是中繼憑證的強大之處。** 如果 K8s 的中繼憑證，和公司外部傳統 VM 的憑證，都是由同一個「企業總部 Root CA」衍生出來的。那麼 K8s 裡面的 Pod，就可以直接拿著自己的 Client Cert，去跟外部的 Oracle 資料庫進行 mTLS 雙向認證，因為大家追根溯源都認同同一個「祖宗（Root CA）」。

**迷思破解 2：K8s 叢集內依然有一把鑰匙全盤皆輸的問題？**
- 沒錯，這在資安術語叫「爆炸半徑（Blast Radius）」。
- **第一層止血（中繼 CA 的意義）**：如果 K8s 的 `ca.key` 被偷了，駭客的確可以毀了這座 K8s 叢集。但因為這只是一張「受限的中繼憑證」，駭客無法用它去偽造公司 HR 系統或外部網銀的憑證。這叫「斷尾求生」，犧牲一個 K8s 叢集，保全整間公司。
- **終極防禦（外部 KMS/Vault）**：為了連 K8s 叢集都不被毀掉，真正的解法是「**K8s 裡面根本不要放 `ca.key`**」。
  - 實務上，我們會修改 API Server 的設定，讓它喪失「自己蓋鋼印」的能力。當有新的 CSR 申請送來時，API Server 會透過加密 API，呼叫外部的 HashiCorp Vault（一個專門保護密碼與金鑰的超級保險箱軟體）。由 Vault 在安全的外部環境蓋完章後，再把憑證傳回給 K8s。駭客就算把 K8s 的主機硬碟整個拔走，也找不到任何 `ca.key`！

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
雖然外部 Vault 架構超出了 CKA 範圍，但在考試中，我們依然要懂得如何利用 K8s 內建的 CA 來模擬這個「簽發（蓋章）」的過程。

```bash
# 🎯 考場神技：K8s 內建的 CSR 審核流程 (模擬你就是那個蓋章的 CA)
# 1. 檢查目前叢集內有哪些待審核的申請表 (Pending 狀態)
kubectl get csr

# 2. 檢視該申請表的詳細內容 (確認身分是否合法)
kubectl describe csr <csr-name>

# 3. 敲下這行指令，等同於呼叫 Controller-Manager 拿出 ca.key 在上面蓋章
kubectl certificate approve <csr-name>

# 4. 提取已經蓋好章的公鑰憑證 (CERT)
kubectl get csr <csr-name> -o jsonpath='{.status.certificate}' | base64 --decode > user.crt
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
- **🎯 考試情境預測：**
  - K8s 官方非常喜歡考 CSR 的流程，因為這證明了你懂 PKI 機制。題目通常會給你一個已經建立好的 `.key` 和 `.csr` 檔案，要求你匯入 Kubernetes 中並 Approve 它，最終讓這個使用者能存取叢集。

- **🛑 避坑指南：**
  - 在將本地端的 `.csr` 檔案內容貼入 YAML 的 `request:` 欄位時，必須經過 Base64 編碼，且**絕對不能有換行符號**。請務必使用 `cat xxx.csr | base64 -w 0` (`-w 0` 代表禁用換行)，否則 API Server 會直接報錯拒絕。

## 2. 📌 核心概念摘要
SSL 是上古時代的加密協定始祖，因存在致命漏洞已遭全球全面淘汰；而 TLS 是 SSL 的正統繼承人。我們現在買的、用的所謂「SSL 憑證」，底層實際跑的全部都是 TLS 協定（目前業界標準為 TLS 1.2 與 1.3）。記住：SSL 已死，TLS 才是現代的標準。

## 4. 🔑 知識點擷取 (Detailed Notes)
**SSL (Secure Sockets Layer) 是什麼？**
- **定義**：1990 年代由 Netscape (網景) 公司發明的加密通道技術。
- **死亡原因**：SSL 3.0 爆發了著名的 POODLE 漏洞，駭客可以輕易破解加密內容。因此，SSL 協定已經在 2015 年被國際資安組織強制宣告死亡 (Deprecated)。
- **業界亂象**：雖然 SSL 協定死了，但「SSL」這個詞已經深植人心。所以現在各大雲端廠商賣的「SSL 憑證」，其實都是「TLS 憑證」，這純粹是行銷與歷史遺留的叫法。

**為什麼叫 TLS (Transport Layer Security)？**
- 當 SSL 3.0 準備升級時，標準化組織 (IETF) 接手了這個項目，為了表示它是開放標準（不再是網景公司的私有財產），於是將其改名為 TLS 1.0。

**TLS 1.0 與 現代 TLS (1.2 / 1.3) 的差異在哪裡？**
- **TLS 1.0 / 1.1 (🛑 已淘汰)**：支援許多老舊、容易被破解的加密演算法（如 MD5、SHA-1）。目前主流瀏覽器（Chrome, Edge）與現代 Kubernetes 已經完全拒絕與 TLS 1.0/1.1 的伺服器連線。
- **TLS 1.2 (✅ 主流標準)**：加入了更強的密碼學套件（如 SHA-256、GCM），是目前絕大多數企業環境與 K8s 預設的最低安全底線。
- **TLS 1.3 (🚀 最新極速版)**：
  - **更安全**：大刀闊斧砍掉了所有舊的、稍有疑慮的加密演算法，只留下最強的幾個。
  - **更快速**：把原本繁瑣的「握手認證過程（Handshake）」從 2 次來回（2-RTT）縮短成 1 次來回（1-RTT），連線速度大幅提升！

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在 K8s 實務中，我們有時需要排查「是不是客戶端太舊，導致無法用 TLS 1.2 連線 API Server？」這時就要用 `openssl` 來強行指定 TLS 版本進行測試：

```bash
# 🎯 考場/實務神技：測試 K8s API Server 是否支援指定的 TLS 版本
# 強制使用 TLS 1.2 嘗試連線 (觀察握手是否成功)
openssl s_client -connect 127.0.0.1:6443 -tls1_2

# 強制使用舊版 TLS 1.0 嘗試連線
# ⚠️ 在現代 K8s 中，這通常會直接被 Server 拒絕並報錯 (Handshake failure)
openssl s_client -connect 127.0.0.1:6443 -tls1

# 使用 curl 測試 API 並印出連線過程中的 TLS 版本資訊
curl -v -k https://localhost:6443
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
- **🎯 考試情境預測：**
  - CKA 考題有時會考驗你對 `kube-apiserver` 啟動參數的熟悉度。題目可能會要求：「提升叢集安全性，確保 API Server 只接受 TLS 1.2 以上的連線。」
  - **解題邏輯**：你需要修改 `/etc/kubernetes/manifests/kube-apiserver.yaml`，在 command 區塊加入參數 `--tls-min-version=VersionTLS12`。

- **🛑 避坑指南：**
  - K8s 預設的 TLS 最低版本就是 1.2。如果你自己開發了一個很舊的 Python 或 Java 應用程式（只支援 TLS 1.0）想去打 K8s 的 API，你會一直收到 `protocol version not supported` 的錯誤。

- **🔧 Troubleshooting：**
  - 當元件無法通訊，日誌顯示 `remote error: tls: protocol version not supported` 或 `handshake failure` 時，這 100% 是因為雙方的 TLS 版本對不起來（通常是一方太舊）。這時請用上面的 `openssl s_client` 指令去探測伺服器到底支援哪些版本。

# Helm 生命週期管理 (Lifecycle Management With Helm)

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 12 節：(2025 Updates) Helm Basics
- **影片標題**：261. Lifecycle Management With Helm

## 2. 📌 核心概念摘要
Helm 的核心價值不僅在於「打包與部署」，更在於「完整的生命週期管理」。本節的底層運作目標是展示 Helm 如何透過 Revision (修訂版本) 機制來精確追蹤每一次的安裝與升級。

這就像是**為 Kubernetes 叢集安裝了「時光機」與「遊戲自動存檔點」**。它讓叢集管理者能夠安全地進行應用程式迭代（Upgrade），並在遭遇非預期錯誤時，擁有迅速且無痛退回（Rollback）穩定狀態的「後悔藥」。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **Revision (修訂版本) 機制**：
  - **定義**：Helm 針對同一個 Release（發佈實例）的每一次配置變更（包含 Install, Upgrade, Rollback），都會自動產生一個遞增的整數 Revision 序號。
  - **底層狀態變化**：最新且成功部署的 Revision 狀態會是 `deployed`；而過去的歷史版本狀態會自動轉為 `superseded`（已被取代）。
- **升級操作 (`helm upgrade`)**：
  - **觸發機制**：當 Chart 釋出新版本，或管理者需要動態修改 `values.yaml` 內的參數（如調高資源限制、更新 Image Tag）時執行。
  - **底層對象變化**：Helm 引擎會比對新舊 YAML 的差異（3-way merge patch），並透過 K8s API 動態更新對應的資源（如只觸發 Deployment 進行 Rolling Update），而非粗暴地全部刪除重建。
- **限制條件 (Limitations)**：
  - **儲存機制**：Helm 3 預設會將這些 Revision 的歷史紀錄序列化並加密，儲存於 Release 所在 Namespace 的 K8s `Secret` 之中。
  - **致命弱點**：若該 Namespace 的相關 Secret 被人為誤刪，Helm 將徹底失去該 Release 的管理、追蹤與回滾能力。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
```bash
# 🎯 考場必備：檢視目前 Namespace 下所有的 Helm Release 狀態與當前版本
helm list

# 🎯 考場必備：追蹤特定 Release 的完整生命週期（找尋可回滾的 Revision 號碼）
helm history nginx-release

# 🚀 實務升級：升級現有的 Release（例如更新 Image 版本）
# 💡 考場救命參數：使用 --install，若該 Release 不存在則直接安裝，存在則升級，避免報錯
helm upgrade nginx-release bitnami/nginx --set image.tag=1.21.4 --install

# 🚑 災難復原：發現新版本崩潰，快速回滾到上一個穩定版本 (例如回到 Revision 1)
# 這是 Helm 最強大的功能之一，能瞬間將 K8s 資源狀態復原
helm rollback nginx-release 1
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
> [!TIP]
> **🎯 考試情境預測**：
> - **題型模擬 1**：考題可能會給你一個已經損壞（CrashLoopBackOff）的 Helm Release，要求你找出它是在哪一次升級壞掉的，並將它安全地 `rollback` 到前一個 Revision。
> - **題型模擬 2**：要求你使用 `helm upgrade` 將某個運作中服務的 Replica 數量提升，或更新其使用的 Chart 版本。

> [!WARNING]
> **🛑 避坑指南**：
> - **Namespace 迷失**：Helm 3 高度依賴 Namespace 隔離。如果你執行 `helm list` 發現空無一物，千萬不要驚慌，請加上 `-n <指定的namespace>` 或 `-A` (所有 Namespace) 來尋找 Release。
> - **Revision 認知錯誤**：在執行 `helm rollback` 時，最後帶入的數字是「Revision 序號」（如 `1` 或 `2`），絕對不是 Chart 的版本號（如 `9.5.13`）。務必先用 `helm history` 確認。

> [!CAUTION]
> **🔧 Troubleshooting**：
> - **狀態卡在 pending-upgrade**：這通常意味著新版本的 Pod 無法順利啟動（例如 Image 標籤打錯導致 ImagePullBackOff）。
> - **除錯路徑**：
>   1. 先用 `helm status <release-name>` 查看 Helm 視角的事件。
>   2. 立刻回歸 K8s 原生除錯：使用 `kubectl get pods -l app.kubernetes.io/instance=<release-name>` 篩選出該 Release 的 Pods。
>   3. 使用 `kubectl describe pod <pod-name>` 檢查 Events，找出資源調度或映像檔拉取的底層錯誤。

## 7. 📝 YAML 骨架 (YAML Skeleton)
Helm 3 是如何儲存 Revision 的？以下是 Helm 在底層建立的 Secret 範例（**強烈警告：絕對不要手動修改它**）：

```yaml
# Helm 3 將每一次的 Revision 儲存為 Secret
apiVersion: v1
kind: Secret
metadata:
  # 命名規則：sh.helm.release.v1.<release-name>.v<revision>
  name: sh.helm.release.v1.nginx-release.v1
  namespace: default
  labels:
    modifiedAt: "1678888888"
    name: nginx-release
    owner: helm
    status: superseded  # 狀態會隨升級自動改變為 superseded 或 deployed
    version: "1"        # 這是 Revision 號碼
type: helm.sh/release.v1
data:
  # 這裡存放了被 base64 加密與 gzip 壓縮的 release 完整資訊 (包含 values 與 templates)
  release: H4sIAAAAAAAA/w...
```

## 8. 🧠 自我測驗
<details>
<summary>如果我一不小心用 <code>kubectl delete secret -l owner=helm</code> 刪除了叢集中的 Helm Secrets，原本透過 Helm 安裝的 Pods 與 Services 會跟著被刪除嗎？我還能使用 <code>helm rollback</code> 嗎？</summary>

**解答：**
1. **不會被刪除**：原有的 Pods、Services 等 K8s 資源會繼續正常運作，因為 Kubernetes Control Plane 已經接管了這些實體資源。
2. **無法 Rollback/Upgrade**：因為 Helm 遺失了該 Release 的所有歷史紀錄（Secret 被刪除），Helm 會認為這個 Release 不存在，你將無法再使用 `helm upgrade`、`helm rollback` 或 `helm list` 來管理它。這也是為什麼備份 Namespace 或操作前確認非常重要。
</details>

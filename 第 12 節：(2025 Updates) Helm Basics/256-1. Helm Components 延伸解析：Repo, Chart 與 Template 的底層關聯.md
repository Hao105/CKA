# Helm Components 延伸解析：Repo, Chart 與 Template 的底層關聯

## 📌 核心觀念摘要
* **真正的核心：渲染引擎 (Rendering Engine)**：Helm 並非強綁定外部網路的工具，它的靈魂在於本機端的「渲染引擎」。這就像是一台**果汁機（合成器）**，能將「配方 (Templates)」與「食材 (Values)」完美攪拌，輸出標準的原生 Kubernetes YAML 交付給 API Server。這使得 Helm 非常適合在 Air-gapped (離線/地端) 環境中使用。
* **Repository (Repo 儲存庫)**：存放大量 Chart 的倉庫，例如公開的 Artifact HUB 或企業私有的 Harbor。這**並非** Helm 運行的絕對必要條件，Helm 完全可以只讀取本地端的資料夾。
* **Templates 與 Values 的化學反應**：`templates/` 目錄下的 YAML 檔案並非標準的 K8s 資源，而是混合了 Go Template 語法 (如 `{{ .Values.xxx }}`) 的半成品。安裝時，引擎會動態讀取 `values.yaml` 的數值，將其注入並替換，完成最終的渲染。

## 💻 必考指令 (Imperative Commands)

在考場或實務中，操作 Repo 與進行參數預覽排錯是必考的基礎連招：

```bash
# 1. 新增遠端 Repo (語法: helm repo add <自定義Repo名稱> <Repo網址>)
helm repo add bitnami https://charts.bitnami.com/bitnami

# 🚨 致命細節：加完 Repo 後必須立刻 update 更新索引，否則找不到套件！
helm repo update

# 2. 地端/離線部署：直接從本地資料夾安裝
# 假設當前目錄下有一個名為 my-chart-folder 的目錄 (裡面包含 Chart.yaml)
helm install my-app ./my-chart-folder

# 3. 考場排錯神器：預覽渲染結果 (Dry Run)
# 不會真的部署到叢集！只會在終端機印出 Helm 組合好的「最終純 YAML」
# 極度適合用來確認 values.yaml 縮排或數值有沒有寫錯
helm install my-app bitnami/nginx --dry-run --debug
```

## 🛠️ 實戰與最佳實踐

> [!WARNING]
> **考試避坑：最常被遺忘的 update**
> 新增 Repo 後直接下 `install`，系統絕對會報錯表示找不到該 Chart。請務必在肌肉記憶中，將 `helm repo add` 與 `helm repo update` 視為必須綁在一起執行的連招。

> [!TIP]
> **SOP：覆蓋參數的兩種方式**
> 當考題要求將某個 Chart 預設的 Service 改成 NodePort 時：
> - **方法一 (快速指令)**：`helm install my-ingress repo/chart --set service.type=NodePort`
> - **方法二 (嚴謹覆寫)**：先建立一個 `custom-values.yaml` 寫入您的設定，再用 `helm install my-ingress repo/chart -f custom-values.yaml`

> [!CAUTION]
> **Troubleshooting 必殺技：Template 渲染失敗**
> **錯誤訊息**：`Error: template: my-chart/templates/deployment.yaml:15: function "xxx" not defined`
> **排查步驟**：這代表在編寫 Template 時，裡面的 Go Template 語法寫錯，或者是變數名稱在 `values.yaml` 裡對不起來。此時請立刻掛上 `--dry-run --debug` 參數重新執行，Helm 會明確指出是哪一行的渲染崩潰，並印出堆疊錯誤。

## 📜 骨架配置 (Template 注入範例)

以下是 `templates/deployment.yaml` 內部如何與 `values.yaml` 產生化學反應的核心骨架：

```yaml
# templates/deployment.yaml (混合了 Go Template 語法的半成品)
apiVersion: apps/v1
kind: Deployment
metadata:
  # 讀取 Helm 內建變數：Release 的名稱
  name: {{ .Release.Name }}-nginx
spec:
  # 讀取 values.yaml 內定義的 replicaCount
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: nginx
          # 讀取 values.yaml 的 image.repository 與 tag，並組合起來
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

## 🧠 自我測驗

<details>
<summary>Q1: 為什麼 Helm 能夠在完全斷網的企業內網 (Air-gapped) 環境中正常部署應用程式？</summary>

**解答：** 
因為 Helm 的核心是本地端的「渲染引擎」。它並不強制依賴外部的 Repository 或網際網路。只要您將包含 `Chart.yaml`, `values.yaml` 和 `templates/` 目錄的資料夾打包帶入內網，Helm 就能直接讀取該本地資料夾，在本地完成 YAML 的渲染並交給 API Server 部署。
</details>

<details>
<summary>Q2: 我剛剛使用 helm repo add 加入了 bitnami 庫，接著立刻下指令 helm install my-db bitnami/mysql，為什麼系統報錯說找不到該 Chart？</summary>

**解答：** 
因為您遺漏了最重要的 `helm repo update` 指令。新增 Repo 只是告訴 Helm 倉庫的網址，必須透過 update 指令將遠端的套件索引清單拉取到本地端，Helm 才會知道這個倉庫裡面究竟有哪些套件名稱與版本可以使用。
</details>

<details>
<summary>Q3: 如果我自己在寫 Chart，想要測試寫出來的範本格式正不正確，但又不想真的把資源建立到 Kubernetes 叢集裡面，該加什麼參數？</summary>

**解答：** 
請在安裝指令後面加上 `--dry-run --debug`。這會讓渲染引擎啟動「模擬執行」模式，並將最終替換合成完畢的純 K8s YAML 直接輸出在終端機畫面上，方便您驗證語法、變數替換與縮排是否皆正確無誤。
</details>

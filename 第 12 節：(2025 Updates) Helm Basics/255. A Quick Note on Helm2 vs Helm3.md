# Helm2 vs Helm3 架構大躍進 (A Quick Note on Helm2 vs Helm3)

## 📌 核心觀念摘要
* **拔除特權黑盒子 (Tillerless)**：舊版 Helm 2 依賴一個名為 Tiller 的 Pod 在叢集中作為伺服器端來建立資源，這通常需要賦予它 `cluster-admin` 等級的最高權限，是嚴重的資安漏洞。Helm 3 將其徹底拔除，轉變為純客戶端架構，直接繼承執行者的權限。
* **狀態一致性革命 (3-Way Merge Patch)**：過去 Helm 2 升級時只會進行「舊版 YAML」與「新版 YAML」的呆板比對 (2-Way)，這會無情覆蓋掉工程師在叢集中的緊急手動修改。Helm 3 引入了「**三方會談**」機制：同時比對舊版、新版與「叢集真實狀態 (Live State)」，精準計算出應有的變更，完美包容了 Sidecar 自動注入與人為緊急修改。
* **去中心化的狀態儲存**：Helm 3 不再把所有的 Release 狀態集中存放在 `kube-system`，而是跟著應用程式，以 Secret 的形式分散儲存在各自的 Namespace 中。

## 💻 必考指令 (Imperative Commands)

在 CKA 考場中現在預設都是提供 Helm 3。您不需要去安裝舊版，但必須熟悉如何驗證與調閱底層資訊來進行排錯：

```bash
# 1. 考場必備：驗證 Helm 版本 (確認為 v3 開頭)
helm version

# 2. 獲取特定 Release 最終渲染出來的 Manifest (排錯神器)
# 當您懷疑 Helm 部署出來的 YAML 跟你想的不一樣，或被 API Server 修改時使用
helm get manifest <release-name> -n <namespace>

# 3. 揭開 Helm 3 儲存狀態的秘密
# Helm 3 將 release 的歷史紀錄存在該 Namespace 的 Secret 裡面
# 尋找 type 為 helm.sh/release.v1 的 Secret 即可證明
kubectl get secrets -n <namespace> | grep helm
```

## 🛠️ 實戰與最佳實踐

> [!WARNING]
> **考試情境預測：RBAC 權限考題**
> 考題可能會要求您先建立一個特定的 ServiceAccount、Role 與 RoleBinding，然後要求您「使用該帳號的權限」去執行 Helm 部署。這就是在考驗您是否理解 Helm 3 **「繼承當前 Client 權限」** 的特性。若權限不足，Helm 部署將直接被 API Server 拒絕。

> [!TIP]
> **SOP：Namespace 隔離機制的避坑**
> 在 Helm 2 時代，所有的 Release 狀態都集中存在 Tiller 所在的 Namespace。但在 Helm 3，Release 的狀態資料是跟著應用程式放在同一個 Namespace 的。
> **SOP：** 若您切換錯 Namespace 下 `helm ls` 指令，會誤以為應用程式沒有部署。永遠記得加上 `-A` 尋找全域，或明確指定正確的 `-n`。

> [!CAUTION]
> **Troubleshooting 必殺技：命名衝突**
> **錯誤訊息：** `Error: INSTALL FAILED: cannot re-use a name that is still in use`
> **排查步驟：** 這代表在該 Namespace 中已經有一個同名的 Helm Release（即便先前的部署失敗了，狀態依然被紀錄在 Secret 裡）。您必須先執行 `helm uninstall <name> -n <namespace>` 清理乾淨，或者使用不同的 Release 名稱。

## 🧠 自我測驗

<details>
<summary>Q1: Helm 3 移除了 Tiller 組件，這對叢集的資安有什麼實質上的幫助？</summary>

**解答：** 
過去 Tiller 為了能幫使用者建立各式各樣的 K8s 資源，通常會被賦予全域的 `cluster-admin` 特權，這是一個極大的安全漏洞。Helm 3 轉為純客戶端架構後，完全依照執行者當下的 `kubeconfig` 權限行事，真正實踐了「最小權限原則」(Least Privilege)。
</details>

<details>
<summary>Q2: 何謂 3-Way Strategic Merge Patch？它解決了 Helm 2 的什麼痛點？</summary>

**解答：** 
這是一種狀態合併機制。Helm 3 在執行 upgrade 時，不僅會比較「舊版 YAML」與「新版 YAML」，還會將「叢集當下的真實狀態 (Live State)」一併納入運算。這完美解決了若有工程師緊急手動修改資源（例如透過 `kubectl edit`），在下次 Helm 部署時不會被粗暴覆蓋掉的問題，也完美包容了自動注入的 Sidecar 容器。
</details>

<details>
<summary>Q3: 如果我想要徹底刪除一個 Helm 部署，但執行 helm ls 卻看不到它，可能的原因是什麼？我要去哪裡尋找它遺留的狀態？</summary>

**解答：** 
最可能的原因是您在錯誤的 Namespace 中。Helm 3 會將部署狀態存成型別為 `helm.sh/release.v1` 的 Secret，並放在與應用程式相同的 Namespace 裡。您可以執行 `kubectl get secrets -A | grep helm` 來揪出它真正的藏身之處。
</details>

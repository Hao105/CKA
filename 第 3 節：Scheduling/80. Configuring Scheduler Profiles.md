# 設定調度器設定檔 (Configuring Scheduler Profiles)

核心介紹 Kubernetes 調度器的工作流、擴展點，以及如何透過單一 kube-scheduler 進程配置多個 Profiles 來實現自定義調度邏輯。

- **調度工作流 (Scheduling Workflow)**： Pod 調度分為四個主要階段：
    - **Scheduling Queue**：依據 PriorityClass 排序（由 `PrioritySort` 處理）。
    - **Filtering**：過濾不符條件的節點（由 `NodeResourcesFit`、`NodeName` 等處理）。
    - **Scoring**：對可用節點評分，選出最佳節點（由 `ImageLocality` 等處理）。
    - **Binding**：正式綁定 Pod 到節點（由 `DefaultBinder` 處理）。
- **擴展點 (Extension Points)**：架構極其靈活，在各階段（Pre-filter, Score, Bind 等）皆有擴展點可插入插件。
- **多設定檔 (Multiple Profiles)**：從 K8s v1.18+ 開始，單一進程可支援多個 Profiles，每個 Profile 運作起來就像獨立調度器。
- **實務目的**：
    - 解決資源競爭 (Race Conditions) 問題：多個 Profile 共享同一個緩存 (Cache)。
    - 針對不同任務自定義邏輯（如 AI 運算）。
    - 降低維護多個二進位檔的維護成本與運算負擔。

### 必考指令

```bash
# 1. 檢查 kube-scheduler 的配置與運行日誌 (診斷調度問題)
kubectl logs kube-scheduler-master -n kube-system

# 2. 建立 Pod 並指定調度器 (使用 dry-run 生成 YAML 模板)
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml

# 3. 在 YAML 的 spec 內手動新增 schedulerName: <profile-name>
# 注意：若未指定則預設為 default-scheduler
```

### YAML 骨架

#### 1. KubeSchedulerConfiguration 範例：定義多個 Profiles
```yaml
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
profiles:
  - schedulerName: default-scheduler  # Profile 1: 預設
  
  - schedulerName: my-scheduler-2     # Profile 2: 自定義插件
    plugins:
      score:
        disabled:
          - name: TaintToleration
        enabled:
          - name: MyCustomPluginA
          
  - schedulerName: my-scheduler-3     # Profile 3: 禁用所有評分階段
    plugins:
      preScore:
        disabled:
          - name: '*'                # 使用 '*' 禁用所有預設插件
      score:
        disabled:
          - name: '*'
```

#### 2. Pod 指定調度器範例
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: special-pod
spec:
  schedulerName: my-scheduler-3      # 必須對應 Configuration 中的名稱
  containers:
  - name: nginx
    image: nginx
```

### 自我測驗

<details>
<summary>Q：為什麼在一個 kube-scheduler 內跑多個 Profiles 比跑三個獨立的調度器二進位檔更好？</summary>

**解答：** 
因為多個 Profile 共享同一個緩存 (Cache)，可以避免多個調度器同時把不同的 Pod 排給同一個節點而造成的資源衝突（Race Condition），且更易於維護與節省運算資源。
</details>

<details>
<summary>Q：在調度階段中，哪一個擴展點負責決定 Pod 在隊列中的先後順序？</summary>

**解答：** 
`QueueSort` 階段。
</details>

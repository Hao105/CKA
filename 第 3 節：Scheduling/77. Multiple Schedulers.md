# 多個調度器 (Multiple Schedulers)

核心介紹如何在 Kubernetes 集群中部署並使用自定義調度器（Custom Scheduler），以滿足預設調度算法無法涵蓋的特殊業務邏輯或檢查條件。

- **高可擴展性**：K8s 允許在保留預設調度器（default-scheduler）的同時，並行運行多個自定義調度器。
- **特定應用需求**：當 Pod 需要基於特殊的自定義條件（而非僅是節點資源或親和性）來選擇節點時，可使用自定義調度器。
- **識別機制**：每個調度器必須有唯一的名稱。若未指定名稱，預設為 `default-scheduler`。
- **部署方式**：現代集群通常將調度器包裝成 **Pod** 或 **Deployment** 部署於 `kube-system` 命名空間中，並透過 ConfigMap 掛載配置。
- **領導者選舉 (Leader Election)**：在高可用 (HA) 架構下，若有多個相同的調度器副本，需開啟此選項確保同一時間只有一個調度器在執行任務。

### 必考指令

```bash
# 1. 檢查調度器 Pod 是否正常運行 (通常在 kube-system 命名空間)
kubectl get pods -n kube-system

# 2. 查看 Pod 的事件，確認是由哪個調度器進行調度
# 使用 -o wide 可以看到更多詳細資訊
kubectl get events -n <namespace> --sort-by='.lastTimestamp'

# 3. 查看自定義調度器的運行日誌 (排錯關鍵)
kubectl logs <my-custom-scheduler-pod-name> -n kube-system

# 4. 驗證 Pod 狀態與細節
kubectl describe pod <pod-name>
```

### YAML 骨架

#### 1. 自定義調度器配置文件 (Kube-Scheduler Config)
通常儲存於 ConfigMap 中，再掛載進調度器 Pod。

```yaml
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
profiles:
  - schedulerName: my-custom-scheduler  # 自定義調度器名稱
leaderElection:
  leaderElect: false                    # 若為單一副本則設為 false
```

#### 2. 讓 Pod 使用特定調度器
在 Pod 的 `spec` 欄位中指定 `schedulerName`。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-custom-pod
spec:
  containers:
  - name: nginx
    image: nginx
  schedulerName: my-custom-scheduler    # 指定使用自定義調度器
```

### 自我測驗

<details>
<summary>Q1：如何確認一個已經運行的 Pod 是由哪個調度器負責放置到節點上的？</summary>

**解答：** 
檢查 Pod 的事件紀錄 (Events)。

**指令：**
```bash
kubectl get events --sort-by='.lastTimestamp'
# 或者查看特定的 Pod 描述
kubectl describe pod <pod-name>
```
在 Events 輸出中尋找 `Scheduled` 類型的事件，其 `From` 欄位會顯示調度器的名稱 (例如 `my-custom-scheduler`)。
</details>

<details>
<summary>Q2：若部署了自定義調度器，但 Pod 一直處於 Pending 狀態且沒有報錯，可能原因為何？</summary>

**解答：** 
1. Pod YAML 中的 `schedulerName` 與調度器配置文件的名稱不匹配。
2. 自定義調度器 Pod 本身故障或未啟動（檢查 `kube-system` 命名空間）。
3. 調度器缺乏必要的 RBAC 權限，無法讀取 Pod 或節點資訊。

**排查指令：**
```bash
# 檢查調度器日誌
kubectl logs <scheduler-pod> -n kube-system
```
</details>

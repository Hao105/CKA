# Kubernetes 核心觀念：Pod 優先級與搶佔 (Priority & Preemption) (H1)

### 核心觀念
*   **優先級定義：** 透過 `PriorityClass` 物件定義權重（數值越大優先權越高，範圍約 -20 億至 20 億）。系統關鍵組件（如控制平面）通常擁有最高權限（約 20 億）。
*   **非命名空間對象：** `PriorityClass` 是集群層級（Non-namespaced）的資源，一旦建立，全集群的 Pod 都能在 spec 中引用。
*   **搶佔機制 (Preemption)：** 當高優先級 Pod 找不到節點調度時，調度器會嘗試終止（Evict）低優先級的 Pod 以釋放資源。
*   **搶佔策略 (PreemptionPolicy)：** 
    *   `PreemptLowerPriority`（預設）：會殺掉低優先級 Pod。
    *   `Never`：不殺掉他人，但在調度隊列中仍排在低優先級 Pod 之前，等待資源釋放。
*   **全局預設：** 可將某個 PriorityClass 設為 `globalDefault: true`，讓未標註優先級的 Pod 自動套用。

### 必考指令 (kubectl)
```bash
# 1. 查看集群中現有的優先級類別 (通常可看到 system-node-critical)
kubectl get priorityclass

# 2. 產出 PriorityClass YAML 骨架 (常用技巧)
# 註：PriorityClass 沒有直接的 kubectl create 命令生成範本，建議記住基本格式或查閱文檔

# 3. 檢查 Pod 目前套用的優先級
kubectl get pod <pod-name> -o yaml | grep priority

# 4. 查看搶佔事件 (若 Pod 被搶佔，會在 Events 中顯示)
kubectl describe pod <pod-name>
```

### YAML 骨架
#### 1. 定義 PriorityClass
```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority-apps
value: 1000000              # 優先級數值
globalDefault: false        # 是否作為全集群 Pod 的預設值
preemptionPolicy: PreemptLowerPriority # 或 Never
description: "這是一個用於關鍵應用程式的優先級類別"
```

#### 2. 在 Pod 中引用
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: critical-app
spec:
  containers:
  - name: nginx
    image: nginx
  priorityClassName: high-priority-apps # 引用上面定義的名稱
```

### 自開測驗 (Toggle list)
*   **情境題 1：如果一個 Pod 沒有在 YAML 中定義 `priorityClassName`，它的預設優先級是多少？如何改變這個行為？**
    *   **解答：** 預設值為 0。若要改變，需建立一個新的 `PriorityClass` 並將屬性 `globalDefault` 設置為 `true`（注意：一個集群只能有一個全局預設優先級）。
*   **情境題 2：什麼情況下高優先級 Pod 不會殺掉（Evict）低優先級 Pod，即使資源不足？**
    *   **解答：** 當該高優先級 Pod 所套用的 PriorityClass 中，`preemptionPolicy` 被設置為 `Never` 時。此時它會在隊列中優先等待，但不會執行搶佔動作。

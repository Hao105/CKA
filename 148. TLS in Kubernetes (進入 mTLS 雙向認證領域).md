## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 7 節： Security

## 2. 📌 核心概念摘要
Kubernetes 內部元件與外部使用者的通訊，全面採用 mTLS (Mutual TLS, 雙向認證) 機制。不僅伺服器（API Server）要向客戶端證明身分，客戶端（如 kubectl、Kubelet）也必須出示自己的憑證與私鑰來證明自己的身分與權限。在 K8s 中，「你的 Client 憑證」就等同於你的「帳號與密碼」。

## 3. 📊 流程圖與視覺化重現 (ASCII / Mermaid)
以下是 K8s 雙向認證（mTLS）的底層交握流程，這就是為什麼雙方都需要 Key 與 Cert 的原因：

```mermaid
graph TD
    subgraph 👤 客戶端 Client (如 admin 或 Kubelet)
        A[擁有: <br> 🔑 client.key <br> 📜 client.crt] 
    end

    subgraph 🏛️ 伺服器端 Server (K8s API Server)
        B[擁有: <br> 🔑 server.key <br> 📜 server.crt]
    end

    A -->|1. 檢查 Server 身分 <br> 驗證 server.crt 的 CA 鋼印| B
    B -->|2. 檢查 Client 身分 <br> 驗證 client.crt 的 CA 鋼印| A
    A -.->|3. 雙方都確認對方是 CA 認可的合法對象| B
    
    C((🌟 K8s 專屬 Root CA)) -.->|核發並信任雙方憑證| A
    C -.->|核發並信任雙方憑證| B
```

## 4. 🔑 知識點擷取 (Detailed Notes)
**為什麼 Client 需要 Key？ (定義與觸發機制)：**
- K8s 預設沒有 users 資料表，不使用傳統的帳號/密碼來登入。
- 當你想以 admin 身分操作叢集時，你必須向 K8s API Server 遞交一張上面寫著 `CN=admin`（Common Name）的 Client Cert (客戶端憑證)。
- 光有憑證不夠，你還必須在本地端擁有對應的 Client Key (客戶端私鑰) 來解開交握過程的加密封包，以此向 API Server 證明「我真的是這張憑證的主人，不是撿來的」。

**雙向認證的底層條件 (Limitations)：**
- Server 端的 `server.crt` 與 Client 端的 `client.crt`，必須都是由同一個 Kubernetes 叢集 Root CA 所簽發的。如果 Client 拿別家 CA 簽發的憑證來，API Server 會直接拒絕連線。

**Kubeconfig 的真實面貌：**
- 我們平常用來連線的 `~/.kube/config` 檔案，裡面其實就是打包了三樣東西：
  - `certificate-authority-data` (用來驗證 Server 的 CA 信任名冊)
  - `client-certificate-data` (你自己的 Client 憑證)
  - `client-key-data` (你自己的 Client 私鑰)

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在考場上，如果你想驗證自己（Client）現在到底是拿什麼 Key 去跟 API Server 溝通，你可以透過以下指令直接透視 Kubeconfig：

```bash
# 🎯 考場神技：查看目前使用的 Kubeconfig 內容
# 加上 --raw 參數，可以把原本被隱藏的 Base64 憑證與金鑰字串全部顯示出來
kubectl config view --raw

# 🔍 實務排錯：將 Kubeconfig 中的 Client Cert 解碼並查看你的「真實身分 (CN/O)」
# (通常考試不會考這麼深，但這是架構師排錯必備)
kubectl config view --raw -o jsonpath='{.users[?(@.name=="kubernetes-admin")].user.client-certificate-data}' | base64 -d | openssl x509 -text -noout | grep "Subject:"
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
- **🎯 考試情境預測：**
  - 考題會給你一個新員工 `john` 的 `.crt` 和 `.key` 檔案。要求你：「建立一個名叫 `john-context` 的 kubeconfig，讓他可以使用這些 Client 金鑰來登入叢集。」
  - **解題邏輯**：使用 `kubectl config set-credentials john --client-certificate=john.crt --client-key=john.key`，將他專屬的公私鑰綁定進設定檔中。

- **🛑 避坑指南：**
  - 在編輯 Kubeconfig 或元件設定時，千萬別把 `ca.crt` (信任清單) 跟 `client.crt` (客戶端身分證) 搞混！`ca.crt` 是用來認證別人的，`client.crt` 是用來證明自己的。

- **🔧 Troubleshooting：**
  - 當你下達 `kubectl` 指令時，如果終端機噴出 `error: You must be logged in to the server (Unauthorized)`：
    - 這代表 API Server 收到了你的連線，但你的 Client Cert 有問題。
    - **優先檢查**你的 Kubeconfig 中，`client-certificate-data` 是否過期，或是 `client-key-data` 是否不匹配。

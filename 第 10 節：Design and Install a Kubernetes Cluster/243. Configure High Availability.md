# 243. Configure High Availability

## 🧠 核心觀念：邁向無中斷服務 (Zero Downtime)
- **避免無腦罷工**：在生產環境中，單一 Master 節點掛掉會導致整個叢集「失去大腦」，無法調度新 Pod 與管理現有資源。
- **高可用性 (HA) 拼圖**：配置 HA 的核心目標，是透過「多個 Master 節點 + 外部負載平衡器 (Load Balancer) + 選主機制 (Leader Election)」這三塊拼圖，確保控制平面具備極強的容錯能力。
- **模式切換**：在 HA 架構中，元件不再是單打獨鬥。API Server 轉為 **Active-Active** 齊心協力，而 Controller 與 Scheduler 則切換為 **Active-Standby** 輪班待命。

## 🔑 核心知識點：三大元件的 HA 配置策略

### 1. API Server (Active-Active)
- **無狀態特性**：API Server 本身不存資料（全部交給 etcd），因此可以同時啟動多個並同時處理大量請求。
- **Load Balancer 依賴**：Worker Node 上的 `kubelet` 以及管理員的 `kubeconfig`，**不能**直接指向某一台 Master 的 IP，**必須**指向前方的 Load Balancer IP。如果 LB 掛掉，即使後端 Master 都健康，叢集依然會全面失聯。

### 2. Controller Manager & Scheduler (Active-Standby)
- **避免腦裂衝突**：為了防止多個 Controller 同時重啟同一個 Pod，或多個 Scheduler 同時把同一個 Pod 亂塞給不同的 Node，這兩個元件只能有「一個」處於活躍狀態 (Leader)。
- **Lease 選主機制 (Leader Election)**：它們會透過 Kubernetes 的 `Lease` (租約) API 物件進行「搶鎖」。誰先搶到鎖就負責工作；其他節點自動轉為 Standby。若 Leader 意外當機沒有續約，Standby 節點會立刻搶鎖接班。

### 3. etcd 的拓撲設計 (Topologies)
- **Stacked Topology (堆疊式)**：etcd 跟 API Server 跑在同一台機器上。優點是省機器成本，缺點是一旦該機器毀損，你會同時失去一個 API Server 和一個 etcd 節點。
- **External Topology (外部式)**：etcd 獨立部署在另外的專屬伺服器叢集上。優點是擁有極致的安全與穩定，缺點是架構複雜、維護成本極高（需要雙倍的機器）。

## 💻 必考實戰指令 (Imperative Commands)

在多 Master 環境中，確認「誰是 Leader」以及「流量入口」是排障的關鍵能力：

```bash
# 👑 1. 查看 HA 叢集中的「選主」狀態 (Leader Election)
# 這能讓你知道目前是哪一台 Master 的 Controller Manager 正在發號施令
kubectl get lease -n kube-system

# 🔍 2. 查看 API Server 負載均衡後的入口 IP
# 檢查 default namespace 中的 kubernetes service，看 Endpoints 是否包含所有健康的 Master IP
kubectl get endpoints kubernetes -n default

# ⚙️ 3. (實務延伸) 檢查 Controller Manager 的 Leader Election 參數
# 查看 static pod YAML 檔中是否有啟用 --leader-elect=true
cat /etc/kubernetes/manifests/kube-controller-manager.yaml | grep leader-elect
```

## 🔧 實戰 SOP 與 Troubleshooting

> [!IMPORTANT]
> **考試情境預測**
> CKA 考試**通常不會**要求你從頭架設 HAProxy + Keepalived (這更偏向 CKS 或 Linux 系統網管的考題)。
> **必考重點**：考題會提供一個已經是 HA 架構的叢集，並要求你進行 `kubeadm upgrade`。
> **升級黃金順序**：先升級第一台 Master 👉 再升級其他 Masters 👉 最後才升級 Workers。

> [!CAUTION]
> **避坑指南：偶數節點的 etcd 死亡陷阱**
> 配置 HA 時，Master 節點（或 etcd 節點）**絕對不要使用偶數** (例如 2 台或 4 台)。
> etcd 的 Raft 演算法需要「超過半數 (Quorum)」同意才能寫入資料。如果是 2 台，掛掉 1 台剩下 1 台 (沒有大於 2/2=1)，整個叢集會進入唯讀狀態，直接死鎖癱瘓！
> **💡 鐵律：標準 HA 節點數配置永遠是奇數 (3, 5, 7 台)。**

> [!TIP]
> **Troubleshooting：為何 kubectl 時好時壞？**
> **症狀**：你發現打 `kubectl` 指令時，有時候會 Timeout 卡住，有時候又秒回。
> **排查方向**：這幾乎可以肯定是前方 Load Balancer 的 Health Check 設定錯誤，或者其中一台 Master 的 `kube-apiserver` 已經當機，但 LB 依然把流量隨機派發給那台死掉的機器。請優先檢查 Load Balancer (如 HAProxy 的 stats 頁面) 的連線狀態。

## 📝 YAML 骨架範例 (Controller Manager Leader Election)

*(實務上可透過觀察 `/etc/kubernetes/manifests/kube-controller-manager.yaml` 中的關鍵參數來確認選主機制是否啟動)*
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: kube-controller-manager
  namespace: kube-system
spec:
  containers:
  - command:
    - kube-controller-manager
    - --allocate-node-cidrs=true
    - --authentication-kubeconfig=/etc/kubernetes/controller-manager.conf
    # ⚠️ 關鍵參數：啟用選主機制，HA 環境預設必須為 true
    - --leader-elect=true
```

## ❓ 自我測驗

<details>
<summary>在一個配置了 3 台 Master 的高可用叢集中，如果你發現執行 <code>kubectl get pods</code> 的回應速度「時快時慢，有時甚至會 Timeout」，最可能的原因是什麼？</summary>

**解答：**
最可能的原因是**前方的 Load Balancer (負載平衡器) 發生了異常設定**。
具體來說，可能是 3 台 Master 中有 1 台的 `kube-apiserver` 已經當機，但 Load Balancer 的「健康檢查 (Health Check)」設定不完善，導致它仍然以 Round-Robin 等輪詢方式將 1/3 的流量派發給那台已經死掉的機器，造成指令時而成功、時而逾時。
</details>

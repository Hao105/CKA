# 242. Choosing Kubernetes Infrastructure

## 🧠 核心觀念：地基決定上層建築
- **基礎設施 (Infrastructure) 的重要性**：部署 Kubernetes 叢集的第一步，是決定你的「底層地基」。無論是使用雲端代管服務 (GKE/EKS)、雲端虛擬機 (EC2/GCE)，還是地端的自建虛擬機 (VMs)，都會直接影響到你對叢集的控制權與維護成本。
- **資源分配的藝術**：理解底層實體資源（CPU、記憶體、網路）是如何被 Hypervisor 切割並分配給 Master 與 Worker 節點，是確保叢集穩定運作與後續排障的基石。
- **貼近實戰**：CKA 考試的核心情境是「自建與維護」，因此我們必須熟悉在原生虛擬機 (IaaS) 上從零開始建立與除錯叢集的過程。

## 🔑 核心知識點：四大選擇層級與限制

### 1. 基礎設施的四大選擇層級
- **本地端測試 (Localhost)**：如 Minikube, Docker Desktop, Kind。將所有元件塞進單一 Container 或 VM 中。適合輕量級開發，**但無法練習跨節點排障**。
- **本地端虛擬機 (Local VMs)**：本次課程架構。使用 Vagrant + VirtualBox 建立多台虛擬機（1 Master + 多 Workers）。這是**最貼近 CKA 考場**的練習環境，但高度依賴本機的 RAM 與 CPU。
- **雲端基礎服務 (IaaS)**：如 AWS EC2, GCP Compute Engine。需自己安裝 OS、設定網路並用 `kubeadm` 自行建叢集。擁有最高控制權，是 **CKA 考試的真實情境**。
- **雲端代管服務 (PaaS/Managed K8s)**：如 AWS EKS, GCP GKE。雲端供應商幫你管好 Master Nodes，你只需專注管 Worker Nodes。注意：**CKA 不考這種類型**，因為你沒有權限管理控制平面。

### 2. 課程架構設計考量 (Our Design)
- **1 Master, 2 Workers**：這是一個權衡了「高可用性學習」與「本機硬體資源限制」的黃金比例。能讓你練習 Pod 的跨節點調度 (Scheduling)，又不會讓筆電資源耗盡當機。
- **⚠️ 硬體最低限制 (Limitations)**：`kubeadm` 在初始化 Master Node 時有嚴格的 Pre-flight Checks (預檢)。Master Node **至少需要 2 顆 CPU 以及 2GB 以上的記憶體**，否則初始化會直接報錯並拒絕執行。

## 💻 必考實戰指令 (Imperative Commands)

```bash
# 🔍 查看叢集中所有節點的「底層基礎設施資訊」
# 包含作業系統版本 (OS Image)、核心版本 (Kernel) 以及容器運行時 (Container Runtime)
kubectl get nodes -o wide

# 🛠️ [自建 K8s 必備] 關閉作業系統的 Swap (虛擬記憶體)
# Kubernetes 強烈要求關閉 Swap，否則 kubelet 會拒絕啟動，導致 Node 變成 NotReady
sudo swapoff -a
# 實務中，記得要去註解掉 /etc/fstab 中的 swap 掛載，否則重開機又會失效！

# 🚀 [練習用] 強制忽略硬體不足的檢查
# 考場上用不到，但你在本機練習資源不夠時能救命
kubeadm init --ignore-preflight-errors=NumCPU,Mem
```

## 🔧 實戰 SOP 與 Troubleshooting

> [!IMPORTANT]
> **考試情境預測**
> CKA 考場的基礎設施**已經由官方準備好**（通常是多個 Ubuntu VMs 組成的叢集）。你不需要從零開始開虛擬機或設定 SSH 密鑰。
> 考題會要求你進行「節點維護 (Node Maintenance)」，例如將某個 Worker Node 升級，或將其標記為不可調度並驅逐上面的 Pod (`cordon` & `drain`)。

> [!CAUTION]
> **避坑指南：迷失在眾多節點中 (Context & Node 錯亂)**
> 這是 CKA 考生最常犯的**致命錯誤**！考場提供**多個叢集** (Clusters)，每個叢集有**多個 Nodes**。
> 當題目要求你去修復 `cluster2` 的 `worker-node-1` 時：
> 1. **必須先切換 Context**：`kubectl config use-context cluster2`
> 2. **必須 SSH 進入正確的機器**：`ssh worker-node-1`
> ⚠️ **嚴重警告**：如果你在 Master Node 上執行了應該在 Worker Node 執行的指令，或者跑到錯誤的叢集，該題**直接 0 分**。下指令前，永遠先看一眼命令提示字元 (Prompt) 的主機名稱！

> [!TIP]
> **Troubleshooting：剛建好的 Nodes 全是 NotReady？**
> **現象**：當你剛建好基礎設施，發現所有 Nodes 用 `kubectl get nodes` 看都是 `NotReady` 狀態，且 CoreDNS 狀態卡在 `Pending`。
> **解法**：這是正常的！剛裝好的 K8s 叢集預設**沒有**安裝 CNI (網路外掛，如 Weave, Calico, Flannel)。只要 Apply 一份 CNI 的 YAML 檔，節點就會瞬間轉為 `Ready`。

## ❓ 自我測驗

<details>
<summary>在 CKA 考場上，如果你正在處理題目要求的 <code>k8s-cluster-3</code> 叢集，但在執行 <code>kubectl</code> 指令前忘記切換 Context，可能會發生什麼嚴重的後果？</summary>

**解答：**
你會把設定或修改套用到「預設的叢集（或者是上一題使用的叢集）」上。這不僅會導致你當前這題拿不到分數，更可能破壞了其他叢集的環境，連帶影響其他題目的得分。
**💡 考試口訣：每做一題，第一件事絕對是先複製題目提供的 `kubectl config use-context ...` 指令並執行！**
</details>

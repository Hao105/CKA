# 285. Troubleshooting - Section Introduction (故障排除戰略總覽)

## 🎯 核心觀念

- **「醫生問診」漏斗法**：故障排除在生產環境與 CKA 考場中是決定架構師身價的關鍵技術。排查的核心目標是建立「由內而外、由上至下」的邏輯漏斗，像醫生問診一樣層層縮小範圍，絕不盲目猜測。
- **層級一：應用程式故障 (Application)**：Pod 處於崩潰狀態 (CrashLoopBackOff) 或 Service 無法正確導流。此時叢集大腦（API Server）是好的，應優先從 Client-side 透過 `kubectl` 撈取第一手日誌與事件。
- **層級二：控制平面故障 (Control Plane)**：Master 節點核心元件故障，最大特徵是 API Server 失聯/逾時。因為核心元件多以 Static Pod 運行，排查重點落在 `/etc/kubernetes/manifests` 與相關憑證配置。
- **層級三：工作節點故障 (Worker Node)**：節點狀態顯示為 `NotReady`。通常是 `kubelet` 服務掛掉、憑證過期，或者是底層容器運行環境 (`containerd`) 停止，此時高度考驗 Linux `systemd` 的指令排查能力。
- **層級四：網路架構故障 (Network)**：跨節點 Pod 無法通訊、DNS 無法解析等。涉及 CNI 插件狀態、`kube-proxy` 的 iptables 規則與 `CoreDNS` 運作是否正常。

## 💻 必考實戰指令 (黃金工具箱)

進入後面實驗前，這五行核心排查指令必須形成肌肉記憶：

```bash
# 1. 🔍 全域事件大搜查 (考場神將)
# 依時間先後順序排列叢集最新事件，讓你一眼看出哪個組件正在報錯
kubectl get events -A --sort-by='.metadata.creationTimestamp'

# 2. 🚨 診斷單一物件的詳細歷史事件與配置錯誤
kubectl describe pod <pod-name> -n <namespace>

# 3. 📝 撈取崩潰容器的「上一世」日誌
# -p (--previous) 參數是排查 CrashLoopBackOff 的致命武器，能看到容器自殺前的最後遺言
kubectl logs <pod-name> -n <namespace> -p --tail=50

# 4. 🧠 Master/Worker 節點底層服務日誌排查 (當 kubectl 失效時使用)
# 必須 SSH 進入該節點執行，-u 指定服務，-f 即時監控，-n 100 顯示最後 100 行
journalctl -u kubelet -n 100 -f

# 5. 🐳 檢查底層容器運行時狀態
systemctl status containerd
```

> [!CAUTION]
> **考場陷阱：切換 Context 與節點迷航**
> 考試時，排查 Control Plane 或 Node 故障通常需要 `ssh` 到特定節點。修復完成後，**務必記得打 `exit` 退回原本的 master/jumpbox 節點**！很多考生在 Worker Node 上直接敲下一題的指令，導致大卡關或指令全部失效。

> [!WARNING]
> **切忌盲目重啟 (Reboot)**
> 遇到問題時，直接重啟 Linux 節點在 CKA 環境中極度危險，可能導致靜態組件徹底死鎖。一律遵循「先看日誌找證據，再對症下藥」的原則。

> [!TIP]
> **排查核心口訣**
> **「先看 Status，再看 Describe，最後看 Logs」**。如果 `kubectl` 全面癱瘓，請立刻轉身走向作業系統層級（使用 `systemctl` 與 `journalctl`）。這套嚴密的邏輯能保證你在高壓考場上穩拿肥美的 30% 分數！

## 📝 YAML 骨架範例 (模擬故障沙盒)

**CrashLoopBackOff 故障模擬 YAML**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: crashing-pod
spec:
  containers:
  - name: faulty-app
    image: busybox:1.36
    # 🔴 刻意寫錯的執行指令將導致容器啟動失敗並陷入無限重啟
    command: ["/bin/sh", "-c", "echo 'I will crash soon'; sleep 2; exit 1"]
```
*利用此 YAML 建立 Pod 後，即可使用 `kubectl logs crashing-pod -p` 練習查看其上一世日誌。*

## 🧠 自我測驗

<details>
<summary>考場情境：你發現某個 Worker Node 的狀態變為 <code>NotReady</code>，同時你無法在叢集內直接用 kubectl 看出明確原因。你會採取哪兩步具體的作業系統層級指令來追查該節點的致命錯誤？</summary>

首先，透過 SSH 登入該問題節點：
```bash
ssh <node-name>
```

登入後，使用以下 Linux 指令進行底層排查：
1. **檢查核心 `kubelet` 狀態**：
```bash
systemctl status kubelet
```
2. **調閱 `kubelet` 錯誤日誌尋找崩潰原因**：
```bash
journalctl -u kubelet -n 100 --no-pager
```
3. **確認底層容器運行環境是否存活**：
```bash
systemctl status containerd
```

排查修復結束後，**務必**執行 `exit` 指令回到原本的操作機台。
</details>

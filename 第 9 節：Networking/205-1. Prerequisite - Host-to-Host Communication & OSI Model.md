# 205-1. Prerequisite - Host-to-Host Communication & OSI Model

## 📌 核心觀念
- **底層網路 (Underlay) 與封裝技術**：主機與主機之間的通訊依賴嚴格的分層協定 (OSI 模型)。在 Kubernetes 叢集中，節點 (Node) 之間的實體通訊構成了底層網路；而 K8s 的 CNI 網路外掛，則是利用「套娃」般的封裝技術 (如 VXLAN 或 IPIP)，將 Pod 的虛擬封包「塞入」標準的 OSI Layer 3/Layer 4 實體封包中，藉此實現跨主機的容器通訊。
- **由上往下封裝，由下往上解封**：在考場上排查網路問題時，必須在腦海中建立一張資料傳遞的透視圖。封包傳送就像「寄包裹」，必須一層一層包裝（加上 Port、IP、MAC），對方收到後再一層一層拆開核對。

## 📊 OSI 模型資料封裝工作流
在排查網路問題時，您必須清楚資料是如何「由上往下封裝」再「由下往上解封裝」的：
```mermaid
graph LR
    subgraph 🖥️ Node A (發送端 192.168.1.10)
        A7[L7 Application: HTTP/API 請求] --> A4[L4 Transport: 加上 TCP/UDP Port]
        A4 --> A3[L3 Network: 加上來源/目的 IP]
        A3 --> A2[L2 Data Link: 加上 MAC 位址]
        A2 --> A1[L1 Physical: 轉為電氣訊號]
    end

    A1 -.-> |🌐 實體交換器/路由器| B1

    subgraph 🖥️ Node B (接收端 192.168.1.11)
        B1[L1 Physical: 接收訊號] --> B2[L2 Data Link: 核對 MAC 位址]
        B2 --> B3[L3 Network: 核對 IP 位址與路由]
        B3 --> B4[L4 Transport: 轉發至對應 Port]
        B4 --> B7[L7 Application: 處理 HTTP 請求]
    end
```

## 🔑 知識點擷取 (Detailed Notes)
我們不需要死背枯燥的七層定義，只需要聚焦與 K8s 網路排錯**絕對相關**的核心層級：

- **L2 資料連結層 (Data Link Layer)**
  - **核心物件**：MAC 位址、Switch (交換器)、ARP 表。
  - **K8s 實務意義**：同一個網段內的 Node 或同一個 Node 內的 Pod 通訊，依賴 L2 的 MAC 位址尋址。如果 ARP 表壞掉，IP 就無法對應到實體網卡。
- **L3 網路層 (Network Layer)**
  - **核心物件**：IP 位址、Router (路由器)、Routing Table (路由表)。
  - **K8s 實務意義**：跨網段的 Node 通訊，或是跨 Node 的 Pod 通訊，都必須經過 L3 路由。CNI 的核心工作之一，就是幫忙在 Linux 核心中寫入正確的 L3 路由規則 (`ip route`)。
- **L4 傳輸層 (Transport Layer)**
  - **核心物件**：TCP/UDP 協定、Port (通訊埠)。
  - **K8s 實務意義**：Service 的 `NodePort` 就是在這一層運作（預設範圍 30000-32767）。此外，Kubelet (Port `10250`) 與 API Server (Port `6443`) 的連線也在這層，若 Port 被防火牆阻擋，叢集將直接崩潰。
- **L7 應用層 (Application Layer)**
  - **核心物件**：HTTP/HTTPS、DNS。
  - **K8s 實務意義**：K8s 的 `Ingress` 就是 L7 的負載平衡器，它可以根據 HTTP 的 URL 路徑 (`/api/v1`) 或主機名稱 (`example.com`) 來決定將封包精準送給哪一個 Service。

## 💻 必考實戰指令
當您遇到「Node NotReady」或「跨節點無法通訊」時，請依照 OSI **由下往上 (L2 ➡️ L3 ➡️ L4)** 進行排查：
```bash
# [L2 測試] 檢查 ARP 表，確認是否有對方的 MAC 位址
arp -n

# [L3 測試] 檢查 IP 與路由是否通暢 (最基礎的 PING)
# 如果 ping 得通，代表 L1~L3 都是正常的！
ping 192.168.1.11

# [L3 排查] 檢查本機路由表，確認封包該往哪個網卡送
ip route

# [L4 排查] 檢查指定的 Port 是否有被監聽 (例如查 Kubelet 10250 是否活著)
ss -tuln | grep 10250

# [L4 測試] 🎯 考場神技：使用 nc (netcat) 測試對端機器的指定 Port 是否開放
# 比 ping 更精準，因為有時防火牆防 ping 但放行 TCP
nc -vz 192.168.1.11 6443
```

## ⚠️ 實戰/最佳實踐 SOP 與 Troubleshooting

> [!TIP]
> **SOP：考試情境預測與避坑指南**
> - **考點通常藏在 L4**：考題要求修復一台 `NotReady` 的 Node，當您登入該 Node 時發現網路介面正常。這時通常是 `kubelet` 服務沒有啟動（導致對應的 Port 沒開），或者是防火牆 (iptables/firewalld) 擋住了 API Server 的通訊埠 (`6443`)。
> - **混淆 L3 與 L4 的排錯工具**：很多學員遇到 `curl` 失敗時，只會一直 `ping`。請記住，`ping` 只能證明 L3 (IP 路由) 沒問題；如果對方機器的 Service 沒在監聽對應的 TCP Port (L4)，ping 會通，但連線依然會被拒絕 (`Connection Refused`)。
> - **Overlay vs Underlay 混淆**：不要把 Node 的 IP (Underlay) 跟 Pod 的 IP (Overlay) 搞混。Pod IP 是虛擬的，離開 Node 之前，CNI 會將 Pod IP 封裝進 Node IP 的實體封包中。

> [!WARNING]
> **Troubleshooting 技巧：跨節點服務存取失敗**
> 情境：在 Node A 執行 `curl http://<Node-B-IP>:<NodePort>` 失敗。
> 1. **排查步驟 1 (測 L3)**：先 `ping <Node-B-IP>`。如果不通，檢查實體網路與路由 (`ip route`)。
> 2. **排查步驟 2 (測 L4)**：如果 ping 通了，請使用 `nc -vz <Node-B-IP> <NodePort>`。如果出現 `Connection refused`，檢查該 NodePort Service 是否真的建立成功，且背後有沒有對應的 Endpoints (Pod) 正常運行。
> 3. **排查步驟 3 (檢查防火牆)**：確認 Node B 的 `iptables` 規則或雲端安全性群組 (Security Group) 是否阻擋了該 NodePort 範圍。

## 📝 YAML 骨架 (L4 NodePort Service)
針對 L4 傳輸層的實作，最典型的 K8s 資源就是 `NodePort` 類型的 Service。考場上若需要快速對外開通 L4 Port 測試連通性，可使用此骨架：
```yaml
apiVersion: v1
kind: Service
metadata:
  name: l4-test-service
spec:
  type: NodePort
  selector:
    app: my-app
  ports:
    - protocol: TCP
      port: 80                  # Service 內部通訊用的 Port
      targetPort: 80            # 對應到 Pod 容器內部的 Port
      nodePort: 30080           # 🚨 關鍵：L4 實體 Node 開放的 Port (預設範圍 30000-32767)
```

## 🧠 自我測驗
<details><summary>當我發現無法透過 NodePort 存取服務時，我嘗試 <code>ping</code> 該 Node 的 IP 發現是可以通的。但當我使用 <code>curl</code> 存取特定 Port 時，卻馬上得到 <code>Connection refused</code> 錯誤。請問這是屬於 OSI 哪一層的問題？我該下什麼指令來驗證？</summary>
這是屬於 <b>L4 傳輸層 (Transport Layer)</b> 的問題。<br><br>
<code>Connection refused</code> 代表封包已經成功抵達對方的機器 (因此 L3 的 ping 得通)，但是對方指定的 Port 並沒有任何應用程式 (Kube-proxy/Pod) 在監聽。<br><br>
驗證方式：
請使用 <code>nc -vz &lt;Node-IP&gt; &lt;NodePort&gt;</code> 來測試該 TCP Port 是否確實處於開啟狀態。若未開啟，請檢查 Service 是否正確設定為 NodePort，以及背後的 Endpoints/Pod 是否為 Ready 狀態。
</details>

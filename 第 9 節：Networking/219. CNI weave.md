# 219. CNI weave

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 9 節：Networking
- **影片標題**：219. CNI weave

## 2. 📌 核心概念摘要
Weave Net 作為 Kubernetes 的 CNI 插件，其最大的特色是透過 DaemonSet 的形式無縫部署於叢集所有節點上。它會自動尋找其他節點上的 Weave 代理程式，建立起一個去中心化的 Mesh (網狀) 網路，實現「免配置 (Zero Configuration)」的跨節點 Pod 通訊與 IP 位址分配 (IPAM)。

## 4. 🔑 知識點擷取 (Detailed Notes)
**部署形式 (DaemonSet)：**
- 從截圖第一段指令 `kubectl get pods -n kube-system` 可以發現，`weave-net` 散佈在 node01, node02, node03, master 等所有節點上。這證明了 CNI 是利用 DaemonSet 確保每個主機都有網路代理程式。

**容器組成細節 (2/2 Ready)：**
- 請特別注意截圖中 READY 欄位顯示為 `2/2`。這代表一個 Weave Pod 內部其實運行著兩個 Container（通常是負責主路由的 `weave` 以及負責網路安全策略的 `weave-npc`）。

**啟動參數與 IPAM：**
從截圖下方的 logs 中可以看到兩行關鍵資訊：
- `ipalloc-range:10.32.0.0/12`：這是 Weave 預設保留給整個叢集 Pod 瓜分使用的超大 IP 網段。
- `supplied peer list: [172.17.0.11 ...]`：Weave 自動抓取了叢集各節點的「實體 Node IP」，並利用這些 IP 建立底層的互相連線。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
截圖中老師使用的指令，正是 CKA 考場中排查 CNI 問題的核心起手式：

```bash
# 1. 檢查 CNI Pod 是否在所有節點上健康運行
# 加上 -o wide 可以直接看到哪個 Node 上的網路代理死掉了
kubectl get pods -n kube-system -l name=weave-net -o wide

# 2. 檢視 Weave 內部運作日誌 (排錯必備)
# ⚠️ 架構師提醒：因為 Weave pod 內有 2 個 container，必須用 -c 參數指定看哪一個
kubectl logs <對應的 weave-net-pod-name> -n kube-system -c weave

# 3. 考場救命指令：若修改了節點網路導致 CNI 異常，可強制重啟 DaemonSet 更新狀態
kubectl rollout restart daemonset weave-net -n kube-system
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
**🎯 考試情境預測：**
- 考場通常不會要求你自己撰寫 Weave 的 YAML，但非常喜歡考 Troubleshooting。例如：「Node 3 的狀態是 NotReady，請修復它。」這時你下 `kubectl get pods -n kube-system` 往往會發現 Node 3 上面的 `weave-net` 處於 CrashLoopBackOff 或 Pending。

**🛑 避坑指南：**
- **新手最常犯的錯就是忘記加 Namespace！** 所有 K8s 核心元件（包含 CNI）預設都在 `kube-system` 底下。如果你只打 `kubectl get pods`，什麼都看不到。
- 看到 Pod 狀態是 `1/2` 不要以為是正常的。如果是 `1/2` 代表有其中一個輔助容器崩潰了，網路策略功能將會失效。

**🔧 Troubleshooting：**
- 如果發現 weave-net 無法啟動（日誌顯示 timeout 或 connection refused），優先檢查該實體節點的防火牆 / 安全群組（Security Group）是否忘記開放 **TCP/UDP 6783**（Weave 節點互相溝通的專用 Port），這是實務上高達 80% 網路外掛裝不起來的原因。

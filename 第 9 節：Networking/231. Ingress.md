# 231. Ingress

## 🧠 核心觀念：大門守衛與他的訪客名單
- **相輔相成**：Ingress 的運作由兩個核心角色組成：**Controller** 與 **Resource**。
- **大門守衛 (Ingress Controller)**：它永遠擋在叢集的「最前面」接收外部真實流量。它是一個真實運作的應用程式（如 Nginx）。
- **訪客名單 (Ingress Resource)**：它是守衛隨身攜帶的「路由規則手冊」。它只是一份寫著「把 `/wear` 轉給 `wear-service`」的 YAML 規則。
- **兩者缺一不可**：沒有 Controller，Resource 就是一張無效的 YAML 廢紙；沒有 Resource，Controller 就是一個不知道要把流量往哪送的無頭蒼蠅。

## 🔑 核心知識點：釐清實體與規則

### Ingress Controller (控制器 / 執行者)
- **定位**：在前面。它是叢集對外的大門守衛。
- **定義**：它是一個**真實運行中**的應用程式（通常以 Deployment 或 DaemonSet 形式跑在 Pod 裡，例如 Nginx, Traefik, HAProxy）。
- **觸發機制**：它會不斷監聽 (Watch) Kubernetes API Server，只要發現有人建立了新的 Ingress Resource，它就會立刻將這些規則動態轉譯成自己聽得懂的設定檔（例如自動修改 `nginx.conf`）並熱重載 (Reload)。
- **⚠️ 預設不自帶**：K8s 預設是**沒有**安裝 Ingress Controller 的（這與預設就有的 `kube-controller-manager` 不同）。

### Ingress Resource (資源 / 規則本)
- **定位**：在旁邊（作為配置檔供守衛讀取）。
- **定義**：它只是 Kubernetes 裡的一種 API 物件（`kind: Ingress`），本質就是一段 YAML 設定檔。
- **限制條件**：如果你只 Apply 了 Ingress Resource 的 YAML，但叢集裡沒有安裝 Controller，什麼事都不會發生。

## 💻 必考實戰指令 (Imperative Commands)

在考場上，你要找 Controller 會去查 Pod，要找 Resource 才是查 Ingress：

```bash
# 🔍 1. 尋找 Ingress Controller (找實體)
# Controller 通常會安裝在特定的 Namespace (如 ingress-nginx)
kubectl get pods -n ingress-nginx
# 查看 Controller 是用 NodePort 還是 LoadBalancer 暴露的
kubectl get svc -n ingress-nginx   

# 🔍 2. 建立與查看 Ingress Resource (寫規則)
# 🎯 [考試加速器] 快速生成 Ingress Resource 的 YAML 範本 (避免手刻大份 YAML)
kubectl create ingress my-ingress --rule="wear.my-online-store.com/wear*=wear-service:80" --dry-run=client -o yaml > ingress.yaml

# 查看叢集內現有的路由規則
kubectl get ingress
kubectl describe ingress my-ingress
```

## 🔧 實戰 SOP 與 Troubleshooting

> [!IMPORTANT]
> **考試配分重點**
> 考題通常會要求你「建立一個 Ingress Resource」將特定網域導向特定的 Service。
> 考場的 Controller 通常已經幫你裝好了，你只需要專注寫出完全正確的 Ingress Resource YAML。注意 `pathType` (通常為 `Prefix` 或 `Exact`) 絕對不能漏掉。

> [!WARNING]
> **避坑指南**
> - **Namespace 隔離陷阱**：Ingress Resource 必須跟它要導向的 Service 放在**同一個 Namespace**！你不能在 `default` 寫一個 Ingress 去指路給 `kube-system` 的 Service。
> - **Annotations 遺漏**：有時候即使你規則寫對了還是回傳 `404`，是因為你沒有告訴 Controller 要怎麼處理路徑（例如 Nginx 常見的 rewrite 陷阱）。如果題目有要求 URL 重寫，記得加上 `nginx.ingress.kubernetes.io/rewrite-target: /`。

> [!TIP]
> **Troubleshooting 排障兩大心法**
> 1. **流量進不來？** 先 `kubectl describe ingress <name>` 檢查你的規則有沒有成功配對到 Service 的 Endpoints (查看 Backends 欄位是否為空)。
> 2. **日誌會說話！** 去看 Controller 的日誌：`kubectl logs -n <controller-namespace> <controller-pod-name>`。如果你的 YAML 有語法錯誤或無法解析的 Annotation，Controller 的日誌會大聲報錯說它無法 Reload 規則。

## 📝 YAML 骨架 (Ingress Resource 範例)

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    # 如果考題要求 rewrite 才需要加
    nginx.ingress.kubernetes.io/rewrite-target: / 
spec:
  rules:
  - host: wear.my-online-store.com
    http:
      paths:
      - path: /wear
        pathType: Prefix
        backend:
          service:
            name: wear-service
            port:
              number: 80
```

## ❓ 自我測驗

<details>
<summary>如果我成功透過 <code>kubectl apply -f ingress.yaml</code> 建立了一個 Ingress Resource，但叢集內尚未安裝任何 Ingress Controller，當外部使用者連線時會發生什麼事？</summary>

**解答：**
什麼事都不會發生，連線會失敗或 Timeout。
因為 Ingress Resource 只是一份「規則清單」，必須要有實體運作中的 Ingress Controller 去讀取這份清單並負責接收與轉發流量。
</details>

# 208-1. 跨主機通訊大解密：Switching, Routing 與 iptables 指令速查與應用場景

## 📌 核心觀念
- **完美的接力賽**：在 Kubernetes 的跨主機通訊中，封包的旅程是一場完美的接力賽。首先透過 Switching (虛擬網橋) 離開 Namespace，接著依靠 OS 層級的 Routing (路由表) 決定前往哪台實體主機，最後經過 iptables 進行安全過濾或必要的 NAT 轉換，才送入實體網卡。
- **CNI 的使命**：CNI (如 Flannel, Calico) 的核心任務，就是自動化地幫您在每一台節點上動態寫入這些底層的路由與防火牆規則。
- **K8s 的鐵律 (無 NAT 跨主機)**：在 K8s 規範中，Pod 對 Pod 的跨主機通訊 **嚴禁使用 NAT**，以確保來源 IP 保持真實；iptables 在此場景主要負責 FORWARD 放行與 Service 的流量轉發。

## 🔑 知識點擷取 (Detailed Notes)

- **L2 Switching (交換) - 場景：同主機 / 離開孤島**
  - **定義**：利用 veth pair 與虛擬網橋 (`cni0` / `docker0`) 處理同網段內的封包交換。
  - **觸發機制**：只要封包的目的地 IP 與自己屬於同一個 Subnet，封包就不會去查路由表，而是直接透過 ARP 廣播尋找 MAC 位址，並由 Switch 進行內部交換。

- **L3 Routing (路由) - 場景：跨主機 / 跨網段找路**
  - **定義**：當封包發現目的地 IP 不在自己的網段時，就會丟給 Gateway (通常是虛擬網橋的 IP)，接著進入 Host 機器的 OS 路由表。
  - **K8s 底層變化**：CNI 會在 Host 上動態寫入路由。例如 Node A 上會有一條規則：「想去 `10.244.2.0/24` (Node B 的 Pod 網段) 的封包，請全部丟給 `192.168.1.11` (Node B 實體 IP)」。

- **iptables (防火牆與 NAT) - 場景：規則過濾與對外通訊**
  - **FORWARD 鏈**：Linux 預設可能出於安全阻擋轉發封包，CNI 必須在 iptables 寫入規則，允許來自 `cni0` 的封包穿過實體網卡。
  - **POSTROUTING (MASQUERADE)**：僅限於當 Pod 要連線到 K8s **叢集外部**（例如 Internet 或實體資料庫）時，才需要做 SNAT (來源位址偽裝)，讓外部網路知道如何把封包送回 Host 主機。

## 💻 必考實戰指令
這是一組「架構師排錯標準起手式」，依照網路傳輸順序整理：
```bash
# ==========================================
# 🛑 場景一：確認 Namespace (Pod) 內部網路狀態
# ==========================================
# 快速列出所有 Namespace，尋找目標
ip netns list

# 進入特定的 Namespace 查看其 IP 與網卡 (模擬在 Pod 內下指令)
ip netns exec <namespace-name> ip a

# 進入 Namespace 查看其預設路由 (通常會指向外面的虛擬網橋 IP)
ip netns exec <namespace-name> ip route

# ==========================================
# 🛑 場景二：確認 Host 主機的 Routing (跨節點關鍵)
# ==========================================
# 1. 檢查主機的核心轉發功能是否開啟 (必備！若為 0 則封包出不去)
cat /proc/sys/net/ipv4/ip_forward

# 2. 🎯 考場神技：檢查主機路由表，尋找 CNI 寫入的跨節點路由
ip route
# 正常運作下，您應該要看到類似這種規則，代表 CNI 有正確寫入路徑：
# 10.244.2.0/24 via 192.168.1.11 dev eth0

# ==========================================
# 🛑 場景三：確認 Host 主機的 iptables (過濾與 NAT)
# ==========================================
# 1. 檢查是否允許封包轉發 (FORWARD chain 預設策略應為 ACCEPT 或有放行規則)
iptables -L FORWARD -n -v

# 2. 檢查 NAT 規則，確認 Pod 連外網的偽裝 (MASQUERADE) 是否存在
iptables -t nat -L POSTROUTING -n -v
```

## ⚠️ 實戰/最佳實踐 SOP 與 Troubleshooting

> [!TIP]
> **SOP：考試情境預測與避坑指南**
> - **考試情境**：在叢集中新增了一台 Worker Node，但發現舊 Node 上的 Pod 始終無法 ping 通新 Node 上的 Pod。
> - **核心考點**：考官可能故意把新 Node 上的 CNI DaemonSet 停掉，或是故意在舊 Node 的作業系統中下了錯誤的 `ip route` 或關閉了 `ip_forward`。
> - **不要被 Docker 慣性思維誤導**：在原生 Docker 中，跨主機通訊預設是不通的（除非做 Port Forwarding 或 Swarm Overlay）。但在 K8s 中，**跨主機的 Pod 必須直接透過 IP 互通且禁止 NAT**。排錯時，請關注 `ip route` 裡的 CNI 規則，而不是去找 Port 映射 (Port Mapping)。

> [!WARNING]
> **Troubleshooting 技巧：跨節點斷線排查邏輯樹**
> 1. **同節點 Pod 互 ping**：如果通，代表 L2 Switching 與虛擬網橋沒壞。如果不通，通常是 CNI 外掛崩潰導致 `cni0` 網卡沒建立。
> 2. **跨節點 Pod 互 ping**：如果不通，先去來源主機下 `ip route`，看有沒有通往目標 Pod 網段的路由。
> 3. **檢查 iptables FORWARD 或 NetworkPolicy**：如果在 `ip route` 看到正確路由，但依然 Timeout，去來源主機下 `iptables -L FORWARD -v -n`，看封包是不是被 DROP 掉了 (通常是防火牆設定異常)。如果底層設定都沒問題，那絕對是 K8s 層級的 **NetworkPolicy** 阻擋了流量！

## 📝 YAML 骨架 (NetworkPolicy - 網路斷線真兇)
在排查完全正常的底層網路，但依然無法連線時，往往是因為叢集中設定了阻擋流量的 NetworkPolicy。這是一個考試常考的「預設全部拒絕 (Default Deny All)」骨架，它會讓該 Namespace 的所有跨節點通訊看似無故 Timeout：
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}            # 🚨 空的 selector 代表選取該 Namespace 下的「所有 Pod」
  policyTypes:
  - Ingress
  - Egress
  # 不寫任何 ingress 或 egress 的允許規則，代表預設拒絕所有進出流量！
```

## 🧠 自我測驗
<details><summary>當 Pod A (Node 1) 無法連線至 Pod B (Node 2) 時，我檢查了 Node 1 的 <code>ip route</code>，發現有正確指向 Node 2 實體 IP 的規則；我也檢查了 <code>ip_forward</code> 值為 1，且 <code>iptables FORWARD</code> 策略也沒有被 DROP。一切底層網路看似正常，請問還有什麼 K8s 原生機制可能導致封包被丟棄？</summary>
最可能的原因是 Kubernetes 的 <b>NetworkPolicy (網路策略)</b>。<br><br>
當 CNI (如 Calico 或 Weave) 成功建立底層網路後，K8s 允許我們透過 NetworkPolicy 在 L3/L4 進行更細粒度的防火牆控制。如果目標 Pod B 所在的 Namespace 有設定「預設拒絕存取 (Default Deny)」的 NetworkPolicy，或者沒有特別放行 Pod A 的 IP 或 Label，那麼封包即使成功抵達 Node 2 的底層，也會被 CNI 在 iptables 產生的防護規則給無聲無息地丟棄 (DROP)。
</details>

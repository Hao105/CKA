# 206. Prerequisite DNS

## 📌 核心觀念
- **翻譯官的角色**：DNS (網域名稱系統) 是將人類易讀的名稱（如 `www.google.com` 或 K8s 內的 `my-database`）轉換為機器可讀 IP 位址的核心服務。
- **不可靠的 IP**：在 Kubernetes 這樣 IP 隨時會因 Pod 重啟或節點故障而變動的動態環境中，我們**「絕對不能」**依賴 IP 來進行服務間的溝通，必須透過 DNS 名稱。
- **排錯的基石**：掌握 Linux 系統解析名稱的順序與除錯工具 (`nslookup`, `dig`)，是徹底理解 K8s 內部 CoreDNS 運作原理與未來在考場排錯的絕對前提。

## 📊 名稱解析標準流程
在考場上，當一個 Pod 無法連線到某個 Service 名稱時，請在腦海中跑一次這個「名稱解析標準流程」：
```mermaid
graph TD
    subgraph local_lookup ["🐳 Pod / Linux 系統內部名稱解析機制"]
        A[應用程式發出請求 \n e.g., curl http://db-svc] --> B{1. 檢查本機對應表 \n /etc/hosts}
        B -->|找到紀錄| C(直接回傳 IP)
        B -->|找不到| D{2. 檢查 DNS 伺服器設定 \n /etc/resolv.conf}

    subgraph dns_lookup ["🌐 DNS 解析查詢"]
        D -->|指向 Nameserver| E[發送查詢請求 \n UDP Port 53]

        F -->|回傳 A Record| G(取得目標 IP)
    end
    
    C -.-> H([建立 TCP/IP 連線])
    G -.-> H
```

## 🔑 知識點擷取 (Detailed Notes)
- **本地端解析優先 (`/etc/hosts`)**
  - **定義**：Linux 系統中最原始的靜態名稱與 IP 對應表。
  - **運作機制**：系統在向外部 DNS 伺服器發起查詢前，必定會先檢查此檔案。在早期沒有 DNS 的年代，或是需要強制覆寫 (Override) 特定網域解析時非常有用。
- **DNS 指派設定 (`/etc/resolv.conf`)**
  - **定義**：告訴當前作業系統「當 `/etc/hosts` 找不到時，你該去問哪一台 DNS Server」。
  - **K8s 實務意義**：在 K8s 中，Kubelet 會動態地將 `nameserver <CoreDNS-ClusterIP>` 注入到每個 Pod 的 `/etc/resolv.conf` 中。這就是為什麼 Pod 知道要找 CoreDNS 解析的原因！
- **核心 DNS 紀錄類型 (Record Types)**
  - **A Record (Address)**：最常見的紀錄，將名稱對應到 IPv4 位址。
  - **AAAA Record**：將名稱對應到 IPv6 位址。
  - **CNAME (Canonical Name)**：別名紀錄，將一個名稱指向另一個名稱（例如將 `www.apple.com` 指向 `apple.com.akadns.net`）。
- **CLI 排錯工具 (`nslookup` vs `dig`)**
  - **`nslookup`**：老牌工具，用於快速查詢單一網域的 IP。
  - **`dig`**：進階工具，能提供更詳細的封包資訊、TTL (存活時間) 以及查詢的路由追蹤。是網路工程師排錯的首選。

## 💻 必考實戰指令
在 CKA 考場上，這幾個指令是驗證網路與服務發現是否正常的「照妖鏡」：
```bash
# 1. 檢查當前機器的 DNS Server 是誰 (不管在 Node 還是 Pod 內都可以下)
cat /etc/resolv.conf

# 2. 基礎 DNS 查詢 (測試名稱能不能轉成 IP)
nslookup www.google.com

# 3. 🎯 考場神技：在 K8s 叢集中測試「內部 DNS (CoreDNS)」是否正常運作！
# 啟動一個帶有 nslookup/dig 工具的暫時性 Pod 來測試解析 K8s Service
kubectl run test-dns --image=busybox:1.28 --rm -it --restart=Never -- nslookup kubernetes.default

# 4. (進階) 使用 dig 查看詳細的 DNS 解析過程
dig www.google.com
```

## ⚠️ 實戰/最佳實踐 SOP 與 Troubleshooting

> [!TIP]
> **SOP：考試情境預測與避坑指南**
> - **解題技巧**：考題若要求您找出名為 `my-database` 的 Service 內部 IP，千萬不要去硬背 Service IP (因為它隨時會變)！您應該使用 `kubectl run` 啟動一個 busybox pod，對 `my-database` 執行 `nslookup`，然後把解析出來的 IP 記錄下來。
> - **跨 Namespace (命名空間) 查詢失敗**：這是學員最常踩的坑！在「同一個 Namespace」下，您可以直接 `ping <service-name>`。但如果 Pod 在 `default`，而 Service 在 `dev-env`，您必須使用**完整網域名稱 (FQDN)**，格式為：`<service-name>.<namespace>.svc.cluster.local`。
> - **不要修改 Node 的 `/etc/resolv.conf`**：除非題目明確要求修復主機的對外網路，否則不要隨便動 Node 上的這個檔案，這可能會導致整個節點無法拉取 Docker Image。

> [!WARNING]
> **Troubleshooting 技巧：Pod 內部名稱解析失敗**
> 情境：在 Pod 內執行 `nslookup` 顯示 `NXDOMAIN` 或是 `Server can't find...`。
> 1. **排查步驟 1**：確認您要找的對象真的存在（Service 有沒有拼錯？有沒有寫對 Namespace？）。
> 2. **排查步驟 2**：檢查該 Pod 內的 `/etc/resolv.conf`，確認裡面的 `nameserver` 指向的 IP 是否確實為 CoreDNS 的 ClusterIP (`kubectl get svc -n kube-system kube-dns`)。
> 3. **排查步驟 3**：檢查 CoreDNS 自身是否存活且正常運作 (`kubectl get pods -n kube-system -l k8s-app=kube-dns`)。如果 CoreDNS 掛了或是不斷重啟，整個叢集的名稱解析都會癱瘓。

## 📝 YAML 骨架 (測試專用 DNS 查詢 Pod)
雖然考試中通常用 `kubectl run` 指令，但若需要寫成 YAML，可以使用以下骨架快速啟動一個帶有網路測試工具的 Pod，並注意其 DNS 策略：
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: test-dns
spec:
  containers:
  - name: busybox
    image: busybox:1.28
    command: ['sleep', '3600']
  dnsPolicy: ClusterFirst   # 🚨 關鍵：確保優先使用 K8s 叢集內部的 CoreDNS 進行解析
  restartPolicy: Never
```

## 🧠 自我測驗
<details><summary>我在 <code>default</code> Namespace 部署了一個 Pod，想要連線到位於 <code>database</code> Namespace 下名為 <code>mysql-svc</code> 的 Service。為什麼我直接對 <code>mysql-svc</code> 進行 ping 測試會得到 <code>bad address</code> 錯誤？我該如何修改連線的網址？</summary>
這是一個經典的「跨 Namespace 解析失敗」錯誤。<br><br>
預設情況下，短名稱（如 <code>mysql-svc</code>）只能在「同一個 Namespace」內部互相解析。如果您的 Pod 與目標 Service 不在同一個 Namespace，CoreDNS 會找不到對應的資源。<br><br>
解決方式：您必須使用該 Service 的<b>完整網域名稱 (FQDN)</b> 才能跨 Namespace 存取。正確的連線位址應該改寫為：<br>
<code>mysql-svc.database.svc.cluster.local</code>
</details>

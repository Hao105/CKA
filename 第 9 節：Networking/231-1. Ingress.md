# 231-1. Ingress: Path-based vs Host-based Routing

## 🧠 核心觀念：切路徑還是切網域？
- **「1 Rule 2 Paths」 (Path-based Routing)**：這是「基於路徑」的路由。就像是同一棟購物中心 (單一 Host) 裡的不同專櫃 (URL 後綴)，透過路徑來區分不同的微服務。
- **「2 Rules 1 Path」 (Host-based / Name-based Routing)**：這是「基於主機/名稱」的路由。就像是兩棟完全獨立的建築物 (不同的子網域)，完全依賴網域名稱來引導流量。
- **YAML 結構的本質差異**：在 Kubernetes 中，這兩種架構決定了你的 YAML 檔案中 `rules` 與 `paths` 陣列的「縮排層級」，這往往是手刻 YAML 時最容易發生災難的地方。

## 🔑 核心知識點：YAML 縮排解析

這兩者的差異在底層 YAML 的結構表現得非常明顯：

### 1. 單一網域，多個路徑 (1 Rule 2 Paths)
- **定義**：流量全部打到同一個 `host`，Controller 透過 HTTP 請求中的 URI (路徑) 來決定轉發給哪個 Service。
- **YAML 結構特徵**：在 `rules` 陣列中**只有一個項目** (`- host: ...`)，但在它底下的 `paths` 陣列中會有**兩個項目** (`- path: /wear` 與 `- path: /watch`)。
- **實務情境**：適合單一大型網站底下的不同模組，或 API 閘道器 (如 `/api/v1/` 與 `/api/v2/`)。

### 2. 多個網域，單一路徑 (2 Rules 1 Path)
- **定義**：Controller 透過 HTTP 請求標頭中的 `Host` 欄位來區別流量。
- **YAML 結構特徵**：在 `rules` 陣列中會有**兩個並列的項目**（兩個 `- host: ...`），然後各自擁有自己的 `http` 與 `paths` 結構。
- **實務情境**：適合多租戶架構 (SaaS)，或是同一個叢集內同時架設兩個完全不相干的網站（例如官網與內部後台）。

## 💻 必考實戰指令 (Imperative Commands)

> [!TIP]
> **考試加速器：自動生成骨架**
> CKA 考場上，如果你需要手動刻 YAML 很容易發生縮排錯誤。請善用多個 `--rule` 參數串接，讓 `kubectl` 幫你生成完美的骨架，這能幫你省下至少 3 分鐘的手寫排版時間！

```bash
# 🎯 [1 Rule 2 Paths] - 同網域 (store.com)，不同路徑
kubectl create ingress path-ingress \
  --rule="store.com/wear*=wear-service:80" \
  --rule="store.com/watch*=watch-service:80" \
  --dry-run=client -o yaml > path-based.yaml

# 🎯 [2 Rules 1 Path] - 不同網域 (wear.store.com / watch.store.com)
kubectl create ingress host-ingress \
  --rule="wear.store.com/*=wear-service:80" \
  --rule="watch.store.com/*=watch-service:80" \
  --dry-run=client -o yaml > host-based.yaml
```

## 🔧 實戰 SOP 與 Troubleshooting

> [!IMPORTANT]
> **考試情境解讀**
> 題目通常會給你一個現有的 YAML 或要求你從頭建一個 Ingress。**請仔細看清楚題目關鍵字：**
> - 若寫 *Routing traffic based on path /app1...* 👉 考單一網域多路徑 (1 Rule 多 Path)
> - 若寫 *Routing traffic for subdomain app1.domain.com...* 👉 考子網域區分 (多 Rules)

> [!WARNING]
> **避坑指南**
> - **YAML 陣列縮排地獄**：`rules` 是一個陣列 (YAML 中有 `-`)，`paths` 也是一個陣列 (也有 `-`)。最常見的死法是**把第二個 `path` 的 `-` 縮排對齊錯了**，導致 API Server 無法解析。強烈建議使用上方指令產出範本再來修改。
> - **`pathType` 必填**：目前的 API 版本 (`networking.k8s.io/v1`) 中，`pathType` 是強制要求的。通常考題會要求設定為 `Prefix` (前綴匹配) 或 `Exact` (精確匹配)。

> [!TIP]
> **Troubleshooting：無 DNS 環境測試法**
> 考場上通常**不會有**真實的 DNS 伺服器幫你把 `store.com` 解析到 Ingress Controller。當你想測試你寫好的 Ingress 是否生效時，請使用 `curl` **強制指定 Host Header**：
> ```bash
> curl -H "Host: store.com" http://<ingress-node-port-ip>/wear
> ```
> 只要 Controller 看到這個 Header，就會正確觸發你的 Rule！

## 📝 YAML 骨架對比

### 1 Rule 2 Paths (Path-based)
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: path-ingress
spec:
  rules:
  - host: store.com
    http:
      paths:
      - path: /wear
        pathType: Prefix
        backend:
          service:
            name: wear-service
            port:
              number: 80
      - path: /watch
        pathType: Prefix
        backend:
          service:
            name: watch-service
            port:
              number: 80
```

### 2 Rules 1 Path (Host-based)
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: host-ingress
spec:
  rules:
  - host: wear.store.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: wear-service
            port:
              number: 80
  - host: watch.store.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: watch-service
            port:
              number: 80
```

## ❓ 自我測驗

<details>
<summary>在 CKA 考場中，如果你設定了一個 Ingress 綁定 <code>example.com</code>，但考場環境沒有內部 DNS 伺服器，你可以如何從你的終端機測試這個 Ingress 是否能正確將流量轉發到後端？</summary>

**解答：**
你可以使用 `curl` 加上 `-H` 參數手動偽造 HTTP Host header。
指令範例：
`curl -H "Host: example.com" http://<Ingress-Controller-的-IP-或-NodePort>/`
</details>

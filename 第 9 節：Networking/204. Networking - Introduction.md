# 204. Networking - Introduction

## 📌 核心觀念
- **平坦無縫的通訊網路**：Kubernetes 網路模型（CNI）旨在解決一個終極目標：「讓叢集中所有的 Pod、Node 與 Service 都能夠在不需要 NAT (網路位址轉換) 的情況下，平坦且無縫地互相通訊」。
- **預備知識的重要性**：在深入 K8s CNI 之前，您必須先具備 Linux 路由 (Routing)、網路命名空間 (Network Namespaces) 以及 DNS 的基礎知識。這些技術就像是「建築物的地基與管線」，是 K8s 底層實作網路隔離與互通的基石。

## 🔑 知識點擷取 (Detailed Notes)
雖然這只是導論，但我們必須先掌握這些即將到來的「預備知識」在 K8s 中扮演什麼角色：

- **Linux Routing & Gateway (路由與閘道器)**
  - **定義**：決定網路封包該往哪個網卡送的規則。
  - **在 K8s 的作用**：CNI (如 Weave) 會在 Node 上建立虛擬網橋，並自動寫入 `ip route` 路由表，讓 Node A 知道如何把封包精準送到 Node B 上的 Pod 網段。
- **Network Namespaces (netns)**
  - **定義**：Linux 核心功能，提供網路堆疊的完全隔離（獨立的網卡、IP、路由表）。
  - **在 K8s 的作用**：一個 Pod 其實就是「共享同一個 Network Namespace 的一組容器」。這就是為什麼同一個 Pod 內的容器可以用 `localhost` 互相溝通的原因。
- **Docker Networking vs. Kubernetes Networking**
  - **差異與限制**：傳統 Docker 依賴 `docker0` 網橋與 SNAT/DNAT 來讓容器連外；但 K8s 規範了更嚴格的網路模型：**Pod 與 Pod 之間必須可以直接透過 IP 溝通，禁止使用 NAT**。這項嚴格規範直接催生了 CNI (Container Network Interface) 的誕生。

## 💻 必考實戰指令
在進入複雜的 K8s 網路設定前，熟悉底層 Linux 網路排錯指令是 CKA 考場上的救命稻草：
```bash
# 1. 檢查當前 Node 的實體與虛擬網卡 (考試中網路不通時，先看網卡有沒有起來)
ip link 
ip a

# 2. 檢查 Linux 路由表 (確認 CNI 是否有正確幫你寫入跨 Node 的路由)
ip route

# 3. 🎯 考場神技：起一個臨時的 busybox Pod 來測試叢集內的 DNS 解析與連通性
# --rm 代表退出後自動刪除，--restart=Never 確保只建 Pod 不建 Deployment
kubectl run test-net --image=busybox:1.28 --rm -it --restart=Never -- sh
# 進入 shell 後，可以測試 ping 其他 Pod 的 IP，或 nslookup 某個 Service 名稱
```

## ⚠️ 實戰/最佳實踐 SOP 與 Troubleshooting

> [!TIP]
> **SOP：考試情境預測與避坑指南**
> - **核心配分區**：雖然 CKA 不會考您從頭寫一個 CNI，但必定會考：(1) 找出並修復 CNI 問題。(2) 使用 `NetworkPolicy` 限制網路連線（這是網路章節的大魔王）。
> - **實作考題**：考官可能給您一個 `kubeadm` 剛建好但處於 `NotReady` 的節點。原因通常是**尚未安裝 CNI**。您必須知道如何套用官方或題目提供的 CNI YAML (例如 Weave 或 Flannel) 來啟動網路。
> - **網段衝突問題**：初始化叢集 (`kubeadm init --pod-network-cidr=...`) 時指定的網段，必須與您後來安裝的 CNI (如 Flannel) 預期的網段一致，否則網路絕對不通。

> [!WARNING]
> **Troubleshooting 技巧：Node 一直處於 NotReady 狀態**
> 若您發現節點持續 `NotReady`：
> - **排查步驟 1**：執行 `kubectl get pods -n kube-system`。
> - **排查步驟 2**：尋找名稱帶有 `weave`, `flannel`, `calico` 等 CNI 關鍵字的 Pod。如果它們處於 `CrashLoopBackOff` 或根本不存在，這就是 Node `NotReady` 的主因。
> - **排查步驟 3**：使用 `kubectl logs -n kube-system <cni-pod-name>` 查看日誌，通常會寫明是網段重疊還是權限問題。

## 📝 YAML 骨架 (網路排錯測試 Pod)
此節為導論，主要依賴 CLI 工具測試網路連通性，在此提供考場實用的臨時除錯 Pod YAML 範例：
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: test-net
spec:
  containers:
  - name: busybox
    image: busybox:1.28
    command: ['sleep', '3600']
  restartPolicy: Never
```

## 🧠 自我測驗
<details><summary>當我使用 kubeadm 建立好叢集後，執行 <code>kubectl get nodes</code> 發現所有 Node 的狀態都是 <code>NotReady</code>，最可能的原因是什麼？我應該優先檢查哪個 Namespace？</summary>
最可能的原因是<b>尚未安裝 CNI (Container Network Interface)</b> 或 CNI 啟動失敗。<br><br>
排查步驟：您應該優先檢查 <code>kube-system</code> Namespace。執行 <code>kubectl get pods -n kube-system</code>，若找不到任何網路套件（如 Weave、Flannel 或 Calico）的 Pod，或者它們處於非 <code>Running</code> 狀態，就證明了網路未就緒導致節點 <code>NotReady</code>。
</details>

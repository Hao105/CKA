# 205. Prerequisite Switching, Routing, Gateways CNI in Kubernetes

## 📌 核心觀念
- **底層網路通訊基石**：在正式進入 Kubernetes 複雜的 CNI (容器網路介面) 前，必須先掌握底層 Linux 的交換 (Switching)、路由 (Routing) 與預設閘道器 (Default Gateway) 運作原理。這些基礎是理解 K8s 如何達成「Pod 間跨節點無 NAT 通訊」的核心。
- **跨節點如過海關**：同一個 Node 內的 Pod 通訊就像是「國內班機」（Switching），不需要檢查護照；但跨 Node 的 Pod 通訊就像是「國際航班」，必須透過海關（Gateway）與航線指引（Routing）才能正確抵達目的地。
- **排錯的終極底層邏輯**：未來在 CKA 考場上排查叢集網路斷線、Pod 無法啟動或節點失聯問題時，往往不是 K8s 設定錯誤，而是底層的 Linux 路由表或網卡發生異常。

## 📊 跨節點封包傳遞流程 (Switching & Routing)
在 K8s 中，封包要怎麼從 Node A 的 Pod 送到 Node B 的 Pod？底層就是依賴 Routing 與 Gateway！
```mermaid
graph TD
    subgraph 🏗️ Node A (192.168.1.10)
        P1(🐳 Pod A: 10.244.1.2) -->|1. Switching 內部交換| B1{虛擬網橋 cni0}
        B1 -->|2. 查無同網段, 丟給 Gateway| R1[🐧 Linux 路由表 Routing]
        R1 -->|3. 跨節點轉發| E1([eth0 實體網卡])
    end

    E1 -.->|4. 實體網路路由| Router((🌐 實體 Router / Gateway)) -.-> E2

    subgraph 🏗️ Node B (192.168.1.11)
        E2([eth0 實體網卡]) -->|5. 接收封包| R2[🐧 Linux 路由表 Routing]
        R2 -->|6. 根據 CNI 路由轉發| B2{虛擬網橋 cni0}
        B2 -->|7. Switching 內部交換| P2(🐳 Pod B: 10.244.2.2)
    end
```

## 🔑 知識點擷取 (Detailed Notes)

- **Switching (網路交換)**
  - **定義**：發生在同一個網段（Layer 2）的通訊，設備之間透過 MAC 位址互相尋找。
  - **K8s 底層對象與限制**：在 Node 內部，同一個 Node 上的 Pod 是透過 CNI 建立的「虛擬網橋（如 `cni0` 或 `docker0`）」進行 Switching 互通。限制是無法跨越子網路通訊。
- **Routing (網路路由)**
  - **定義**：發生在不同網段（Layer 3）的通訊，決定封包該走哪一條路徑送達目的地 IP。
  - **K8s 底層對象與限制**：當 Pod 要連線到外部或其他 Node 的 Pod 時，封包會送到 Node 的主機層，這時必須依賴 Linux 的 `ip route` 路由表來決定封包要丟給哪個網卡。如果 CNI 外掛壞掉，通常就是這裡的路由表沒寫入成功。
- **Default Gateway (預設閘道器)**
  - **定義**：當路由表裡找不到特定的目的地 IP 時，封包就會統一丟給「預設閘道器」當作最後的出口。
  - **觸發機制**：任何對外網（如拉取 Image、連線到 8.8.8.8）的請求都會觸發。如果節點的預設閘道器設定錯誤，Node 會直接陷入 `NotReady` 狀態且無法下載任何容器映像檔。
- **Network Namespaces (網路命名空間) 🧠 核心技術**
  - **定義**：Linux 核心的隔離技術，讓每個 Namespace 擁有自己獨立的網卡、IP 位址與路由表。
  - **K8s 應用**：一個 Pod 說穿了就是「一個 Network Namespace」！這也是為什麼每個 Pod 都能有一個獨立的 IP，且彼此的網路環境與 Port 不會發生衝突的底層原因。

## 💻 必考實戰指令
這節課是預備知識，因此指令以「Linux 網路排錯」為主，這些在考場上網路不通時能救你一命：
```bash
# 1. 檢查當前節點有哪些網卡 (包含實體 eth0 與虛擬 cni0)
ip link show

# 2. 檢查當前節點的 IP 位址配置
ip addr

# 3. 🚨 考場神技：檢查 Linux 路由表與預設閘道器 (Default Gateway)
# 當跨節點網路不通，或 Node 無法連外網時，務必下這行檢查！
ip route
# (正常情況下會看到類似： default via 192.168.x.x dev eth0)

# 4. 檢查 ARP 表 (MAC 位址與 IP 的對應，排查同網段 Switching 問題)
arp -a
```

## ⚠️ 實戰/最佳實踐 SOP 與 Troubleshooting

> [!TIP]
> **SOP：考試情境預測與避坑指南**
> - **配分重點**：考題可能會給您一個「壞掉的叢集」，要求您修復一台無法加入叢集的 Worker Node。原因往往是該 Node 的網路介面沒起來，或預設路由被錯誤修改。
> - **只看 K8s 是不夠的**：很多學員遇到網路問題只會瘋狂敲 `kubectl get pods`，如果問題出在底層 Linux 路由，這樣是看不出所以然的。請務必 **SSH 進入出問題的 Node**，使用 `ip route` 或 `ping` 檢查實體網路狀況。
> - **CNI 網段重疊 (IP Overlap)**：在初始化叢集時設定的 Pod 網段（`--pod-network-cidr`）絕對不能跟您機房實體網路（或雲端 VPC）的網段重疊，否則 Routing 會大混亂，封包永遠送不到正確的地方。

> [!WARNING]
> **Troubleshooting 技巧：CNI 建立網路環境失敗**
> 情境：Pod 狀態顯示 `ContainerCreating` 很久，最後事件紀錄噴出 `Failed to create pod sandbox: rpc error... networkPlugin cni failed to set up pod`。
> - **排查步驟 1**：執行 `kubectl get pods -n kube-system`，檢查 CNI Pod 是否存活。
> - **排查步驟 2**：SSH 登入出問題的 Node，執行 `ip route`，確認 CNI (如 Flannel 或 Calico) 有沒有正確建立通往其他 Node 的路由規則。
> - **排查步驟 3**：檢查 Kubelet 日誌 (`journalctl -u kubelet`)，看是否是底層 Linux 網橋 (Bridge) 或 IPTables 模組未正確載入導致網路建立失敗。

## 📝 YAML 骨架 (網路排錯用 DaemonSet)
在排查跨節點路由或 Node 連線異常時，若需要快速驗證每個節點的網路狀態，您可以部署具備 `hostNetwork: true` 的測試用 DaemonSet：
```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: net-troubleshoot-ds
spec:
  selector:
    matchLabels:
      app: net-troubleshoot
  template:
    metadata:
      labels:
        app: net-troubleshoot
    spec:
      hostNetwork: true            # 🚨 關鍵考點：讓 Pod 直接使用宿主機的網路命名空間，方便進行 Node 層級路由測試
      containers:
      - name: busybox
        image: busybox:1.28
        command: ["sleep", "3600"]
```

## 🧠 自我測驗
<details><summary>我在 K8s 中建立了一個 Deployment，但發現裡面的 Pod 卡在 <code>ContainerCreating</code> 狀態，並且報錯 <code>networkPlugin cni failed to set up pod</code>。身為 K8s 管理員，我不能只看 kubectl 指令，下一步我該進行什麼底層檢查？</summary>
這個錯誤訊息明確指出是 CNI 無法為 Pod 建立底層網路環境。正確的底層排查思路應該是：
<br><br>
1. <b>登入出問題的節點</b>：透過 SSH 進入卡在 ContainerCreating 的 Pod 所在的 Node。
2. <b>檢查網路介面與路由</b>：執行 <code>ip link show</code> 與 <code>ip route</code> 檢查虛擬網橋（如 cni0）是否建立成功，以及路由表是否正常。
3. <b>檢查系統服務日誌</b>：執行 <code>journalctl -u kubelet</code> 或查看容器執行階段的日誌，確認是否有作業系統層面缺失模組（例如 CNI 外掛二進位檔遺失或權限不足）的問題。
</details>

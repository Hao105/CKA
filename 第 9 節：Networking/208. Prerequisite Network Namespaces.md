# 208. Prerequisite Network Namespaces

## 📌 核心觀念
- **封閉與隔離的孤島**：Network Namespace (網路命名空間) 是 Linux 核心提供的一種資源隔離技術，它讓特定的行程 (Process) 擁有自己獨立的網卡、IP 位址與路由表。
- **Pod 的網路本質**：在 Kubernetes 中，「一個 Pod 的本質就是一個 Network Namespace」。
- **打破隔離的管線工程**：CNI (如 Flannel/Calico) 的底層工作，其實就是自動化地幫這些封閉的 Namespace 接上虛擬網路線 (`veth pair`)、設定網橋 (`Bridge`) 與 NAT 規則，讓它們能跨節點、跨實體網路互通。

## 📊 封包出網透視圖 (Namespace ➡️ 實體網路)
為了應對 CKA 的網路排錯，您必須清楚封包是如何從「封閉的 Namespace」一路鑽出「實體主機」的：

## 🔑 知識點擷取 (Detailed Notes)

- **隔離與互聯機制**
  - **Network Namespace**：創造出一個完全沒有網路介面（預設只有 loopback）的孤島。
  - **veth pair (虛擬乙太網路對)**：像是一條虛擬的網路線，一端插在 Namespace 裡面，另一端插在 Host 主機上，用來打破隔離。
  - **Virtual Bridge (虛擬網橋)**：如截圖中的 `v-net-0` 或實務上的 `cni0` / `docker0`。它相當於一台 Switch，把多個 Namespace 的 veth pair 全都接在一起，讓它們可以在同一個虛擬網段內互相 ping 通。

- **為什麼連外網需要 NAT (MASQUERADE)？**
  - **觸發機制**：當 Namespace (例如 `192.168.15.2`) 想要連線到外部實體網路 (例如 `192.168.1.3` 或 `8.8.8.8`) 時。
  - **限制條件**：外部路由器並不認識 `192.168.15.0/24` 這個虛擬的私有網段。如果封包直接送出去，回傳的封包會因為外部機器找不到回傳路徑而被丟棄。
  - **解決方案**：透過 `iptables -t nat ... -j MASQUERADE`。當封包準備離開 Host 的實體網卡 (`eth0`) 時，主機會強制將封包的「來源 IP」偽裝成 Host 自己的實體 IP (`192.168.1.2`)。這樣外部機器就能成功將封包回傳給 Host，Host 再查表將其轉交回對應的 Namespace 內部。

## 💻 必考實戰指令
在 CKA 考場上，這組 `ip netns` 指令是您檢查底層容器網路的「透視鏡」：
```bash
# 1. 檢視當前 Linux 主機上有多少個隔離的 Network Namespace
# (注意：Docker/K8s 預設會隱藏 ns，實務上通常透過 nsenter 等工具進入)
ip netns list

# 2. 🎯 模擬進入 Pod 內部執行網路指令
# 語法：ip netns exec <namespace-name> <command>
# 例如：檢查名為 "blue" 的 Namespace 內的路由表
ip netns exec blue ip route

# 3. 檢查主機是否開啟了「IP 路由轉發」功能 (值必須是 1)
# 如果是 0，您的 Pod 絕對連不上外網，因為主機拒絕幫忙轉發封包！
cat /proc/sys/net/ipv4/ip_forward
# 也可以使用 sysctl 查詢：
sysctl net.ipv4.ip_forward

# 4. 檢查 Node 上的 NAT 規則是否有正確建立 MASQUERADE (偽裝)
iptables -t nat -L POSTROUTING -n -v
```

## ⚠️ 實戰/最佳實踐 SOP 與 Troubleshooting

> [!TIP]
> **SOP：考試情境預測與避坑指南**
> - **考題情境**：可能會給定一個無法連外網的 Kubernetes Worker Node。您檢查 CNI 發現運作正常，但 Pod 依然無法 ping 通外網的 `8.8.8.8`。這時考點往往藏在底層的 Linux 設定：可能是主機的 `ip_forward` 被關閉，或者是 `iptables` 規則發生損毀。
> - **嚴禁手動清空防火牆**：很多學員在遇到網路問題時，習慣性地去執行 `iptables -F`（清空規則）。在 K8s Node 上做這件事是**毀滅性**的！這會把 CNI (如 Flannel/Weave) 辛苦建好的 NAT 規則與 Service 的轉發規則全部刪光，導致該節點上的所有 Pod 瞬間斷網。

> [!WARNING]
> **Troubleshooting 技巧：Pod 內部無法連線外部網路**
> 情境：Pod 內部可以互相 Ping 通 (代表虛擬網橋 Bridge 正常)，但卻無法連線到外部網路或外部的資料庫。
> 1. **排查步驟 1 (檢查轉發機制)**：在 Node 主機上執行 `sysctl net.ipv4.ip_forward`，確保主機允許幫忙轉發封包 (數值必須為 `1`)。
> 2. **排查步驟 2 (檢查偽裝機制)**：檢查 iptables。如果 CNI 外掛曾經崩潰，可能會導致 `MASQUERADE` 規則遺失。解決方式是嘗試刪除並重啟 CNI 的 DaemonSet，讓它在啟動時自動幫 Node 重新修復好 iptables 規則。

## 📝 YAML 骨架 (突破 Namespace 隔離的特權 Pod)
在排查網路問題時，有時我們需要讓 Pod 脫離 Network Namespace 的隔離，直接使用主機的實體網路。在考場上若需要部署排錯工具，請記得這個 `hostNetwork` 骨架：
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: netshoot-host-pod
spec:
  hostNetwork: true          # 🚨 關鍵考點：設為 true 代表打破 Namespace 隔離，直接使用 Node 主機的網卡與 IP
  containers:
  - name: netshoot
    image: nicolaka/netshoot # 一款強大的網路除錯工具包
    command: ['sleep', '3600']
```

## 🧠 自我測驗
<details><summary>當我發現同一個 Node 上的兩個 Pod 可以互相 ping 通，但是這兩個 Pod 卻都無法 ping 通外網的 <code>8.8.8.8</code> 時，請問最可能的問題點出在剛剛架構圖中的哪一個環節？我該如何檢查？</summary>
Pod 間可以互通，代表虛擬網橋 (Bridge) 與 veth-pair 的內部網路交換是完全正常的；但封包在嘗試「離開 Host 主機」時失敗了。<br><br>
最可能的問題出在以下兩個環節：
1. <b>IP Forwarding 未開啟</b>：請執行 <code>cat /proc/sys/net/ipv4/ip_forward</code> 檢查數值是否為 1。如果為 0，Linux 核心會直接拒絕為不同網卡進行轉發。
2. <b>NAT (MASQUERADE) 偽裝失敗</b>：如果封包沒被偽裝成主機的實體 IP 就直接送出實體網卡，外網是無法找到路徑回傳給虛擬網段的。請執行 <code>iptables -t nat -L POSTROUTING</code> 檢查是否有產生對應的 SNAT/MASQUERADE 規則。
</details>

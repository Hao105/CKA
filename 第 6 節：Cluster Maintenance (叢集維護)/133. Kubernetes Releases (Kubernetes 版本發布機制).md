## 2. 📌 核心概念摘要
了解 Kubernetes 的版本命名規範 (Semantic Versioning) 與發布週期 (Release Cadence)，是執行叢集安全升級的最重要前置知識。這決定了叢集升級時的「相容性邊界」，並確保核心元件 (如 kube-apiserver 與 kubelet) 之間不會因為版本落差過大而無法溝通。

## 4. 🔑 知識點擷取 (Detailed Notes)
這是 CKA 考試中關於「升級限制」的底層規則，請務必牢記：

- **版本號格式**：採用 `vX.Y.Z` 格式。例如 `v1.32.1`。

- **發布週期 (Release Cadence)**：
  過去 Kubernetes 一年發布 4 次 Minor 版本；近年為了減輕維護者的壓力，已改為一年發布 3 個 Minor 次版本 (例如：1.30, 1.31, 1.32)。

- **官方支援生命週期 (Support Window)**：
  官方通常只會為最新的 3 個次版本提供 Patch (修補) 與 CVE (資安) 更新（大約 14 個月的支援期）。過期的版本將不再獲得官方安全保障。

- **功能成熟度標籤**：K8s 的新功能會經歷三個階段：
  - **Alpha**：預設關閉，可能會有 Bug 或隨時被移除。
  - **Beta**：預設開啟，程式碼已經過測試，但細節可能還會微調。
  - **Stable (GA)**：穩定版本，會出現在標準的版本發布中。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在動手升級叢集前，第一步永遠是檢查當前版本：

```bash
# 💡 考點 1：檢查當前 Client (kubectl) 與 Server (API Server) 的精確版本
# 考試時務必先確認自己在哪個版本，才知道下一步該升級到哪
kubectl version 

# 💡 考點 2：取得叢集中各個 Node 的 Kubelet 版本
# 升級過程中，用來確認 Worker Node 是否已成功升級
kubectl get nodes

# 💡 考點 3：(預告) 檢查哪些版本可以被升級
# 這是下一堂實作課最核心的指令，用來確認升級路徑與相容性
kubeadm upgrade plan
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
🎯 **考試情境預測**：
> 考題通常會直接要求你：「將目前的叢集從 v1.31.1 升級到 v1.32.0」。此時你腦中必須立刻浮現版本升級的相容性規則。

🛑 **避坑指南 (版本偏差政策 Version Skew Policy - 🌟 超級大考點)**：
> - **絕對不可跳級升級 (No Skip Minor Version)**：如果你的叢集目前是 1.30，而考題要求你升到 1.32。你不可以直接升到 1.32！你必須先升到 1.31，再升到 1.32。跨越 Minor 版本升級會導致資料庫 (etcd) 或 API 不相容而毀損。
> - **元件版本階級**：在同一個叢集中，`kube-apiserver` 的版本永遠必須是最高的。
> - **Kubelet 向後相容**：`kubelet` 的版本可以比 `kube-apiserver` 舊，但最多只能落後 3 個 Minor 版本 (N-3)。例如 API Server 是 1.32，Kubelet 最舊可以支援到 1.29 (這也是為什麼我們可以先升級 Control Plane，之後再慢慢升級 Worker Nodes 的底層原因)。

🔧 **Troubleshooting**：
> 若升級後發現 Node 顯示為 `NotReady`，請先使用 `kubectl get nodes` 檢查該節點的 VERSION 欄位。若版本還是舊的，代表該節點的 kubelet 尚未被正確升級或重啟。

# 135. Cluster Upgrade Introduction (叢集升級前導與版本相容政策)

## 1. 🏷️ 課程定位
- **章節編號與名稱**：第 6 節：Cluster Maintenance (叢集維護)
- **影片標題**：135. Cluster Upgrade Introduction (叢集升級前導與版本相容政策)

## 2. 📌 核心概念摘要
Kubernetes 叢集升級絕非「一鍵更新」，它是由多個獨立運作的元件交織而成的分布式系統。本節的核心運作目標是掌握官方的 「版本偏差政策 (Version Skew Policy)」，理清哪些元件可以有版本落差、落差的極限是多少，進而規劃出絕對安全的升級先後順序。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **📌 核心元件 vs 外部附加組件**
  - **核心控制平面 (Core Control Plane)**：包括 `kube-apiserver`、`controller-manager`、`kube-scheduler`、`kubelet`、`kube-proxy`。在一個健康穩定的叢集中，它們最終應該趨於相同的 Minor 版本（例如隨影片範例的 v1.13.4）。
  - **外部依賴 (External Addons)**：`etcd` 核心資料庫與 `CoreDNS` 屬於獨立的開源專案，擁有自己獨立的版本號（如 etcd v3.x），它們的版本相容性由 K8s 官方發布公告指定，通常不需要與 K8s 核心版本號一致。

- **⚠️ 考場必背：版本偏差政策 (Version Skew Policy)**
  這是 CKA 選擇題與實作題判斷「能不能升級」的絕對真理：
  - **kube-apiserver 是天花板**：叢集中沒有任何核心元件的 Minor 版本可以高於 `kube-apiserver`。
  - **Kube-Controller-Manager & Kube-Scheduler**：這兩位是大腦的直屬助手，版本審查最嚴格。它們最多只能比 API Server 慢一個 Minor 版本 (`N-1`)。
  - **Kubelet & Kube-Proxy**：負責現場執行的小工頭，相容性較寬容。它們最多可以比 API Server 慢三個 Minor 版本 (`N-3`)。這也是為什麼我們可以「先升級 Master 節點，過幾天再慢慢升級 Worker 節點」的底層科學依據。
  - **Kubectl (用戶端工具)**：遙控器可以稍微超前或落後。它支援 `N+1`、`N`、與 `N-1`。也就是說，v1.32 的 kubectl 可以安全地去操控 v1.31、v1.32 與 v1.33 的 API Server。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
本節為觀念篇，在動手升級前，必須先使用以下指令撈出所有元件的「當前版本」以比對 Version Skew 是否合規：

```bash
# 💡 實戰技巧 1：同時查看用戶端 (kubectl) 與叢集大腦 (apiserver) 的版本
# 檢查兩者是否符合 N+/-1 的相容規範
kubectl version --short

# 💡 實戰技巧 2：撈出所有 Worker Node 上的 kubelet 精確版本
# 升級前用來確認各節點落後的版本 (N-x) 是否在安全邊界內
kubectl get nodes -o wide

# 💡 實戰技巧 3：查看靜態 Pod (Static Pods) 的 Image 標籤
# 檢查大腦元件 (apiserver, scheduler, controller-manager) 當前運行的版本
kubectl get pods -n kube-system -o custom-columns=NAME:.metadata.name,IMAGE:.spec.containers[*].image
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
🎯 **考試情境預測**：
> **觀念題/選擇題**：給定一個 `kube-apiserver` 版本為 v1.32 的叢集，問你以下哪一個元件的版本配置是非法 (Unsupported) 的？
> (A) `kube-scheduler`: v1.31 (合法，N-1)
> (B) `kubelet`: v1.29 (合法，N-3)
> (C) `kube-controller-manager`: v1.33 (❌ 非法！高於 apiserver)

🛑 **避坑指南 (升級的黃金順序)**：
> 因為 Version Skew Policy 的硬性限制，升級順序絕對不能顛倒。
> **正確順序**：`kube-apiserver` ➡️ `kube-controller-manager` ➡️ `kube-scheduler` ➡️ `kube-proxy/kubelet`。
> 如果你在考場上不小心先去升級了 Worker Node 的 kubelet，導致工頭版本高於大腦，該節點會立刻在叢集中失聯。

🔧 **Troubleshooting (版本不相容的徵兆)**：
> 當你完成 Control Plane 升級後，若發現某些舊版的 kubelet 節點開始頻繁報錯，或無法執行新的 API 功能，請優先檢查該 kubelet 的版本是否落後超過了 3 個 Minor 版本（變成了 N-4）。若是，必須立刻將該節點的 kubelet 依序推進升級。

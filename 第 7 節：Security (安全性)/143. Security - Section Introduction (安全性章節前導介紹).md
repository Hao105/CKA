## 2. 📌 核心概念摘要
在 Kubernetes 中，安全性不是單一的設定，而是一層一層的防護網。本章節的核心目標是徹底理解並實作 Kubernetes 的「存取控制 (Access Control)」與「邊界防護」。我們將探討：你是誰 (Authentication)、你能做什麼 (Authorization)，以及如何保護叢集內部的通訊與應用程式 (Certificates, Network Policies, Security Contexts)。

## 4. 🔑 知識點擷取 (Detailed Notes)
根據本章節的播放清單大綱，以下是您即將面臨的五大安全領域與考點預覽：

1. **身份驗證與憑證 (Authentication & TLS)**：
   Kubernetes 內部所有元件 (API Server, Kubelet, ETCD) 的溝通預設都透過 TLS 加密。您必須學會如何查看、除錯與手動簽發 TLS 憑證 (Certificates API)。

2. **授權與角色存取控制 (RBAC)**：
   這是 CKA 必考大題！您必須精通 Role、RoleBinding (限制於單一 Namespace) 以及 ClusterRole、ClusterRoleBinding (影響整個叢集) 的精確綁定關係。

3. **服務帳號 (Service Accounts)**：
   給予 Pod (機器人) 存取 Kubernetes API 的身分證，通常會與 RBAC 結合考驗。

4. **容器安全上下文 (Security Contexts)**：
   限制 Pod 或 Container 在 Linux 節點上的權限（例如：不可使用 root 執行、不可寫入根目錄、賦予特定 Linux Capabilities）。

5. **網路策略 (Network Policies)**：
   預設情況下，K8s 叢集內的 Pod 可以互相暢通無阻 (All-to-All)。您必須學會撰寫 Network Policy，像防火牆一樣精準阻擋或放行特定 Label 的 Pod 流量。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
*(雖然這是前導課，但這幾個指令是本章節的靈魂起手式，建議先刻進肌肉記憶)*

```bash
# 💡 考點 1：(超級神技) 檢查我或是某個 ServiceAccount 有沒有權限做某件事
# 考試時用來驗證你的 RBAC 有沒有設定成功！
kubectl auth can-i create deployments --as=dev-user
kubectl auth can-i delete pods -n kube-system

# 💡 考點 2：快速建立 ServiceAccount (不要手寫 YAML)
kubectl create serviceaccount my-sa

# 💡 考點 3：快速建立 Role 與 RoleBinding (極速拿分法)
kubectl create role my-role --verb=get,list,watch --resource=pods
kubectl create rolebinding my-binding --role=my-role --serviceaccount=default:my-sa
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
🎯 **考試情境預測**：
> - **RBAC 題型**：「請建立一個名為 john 的 User，並透過 RBAC 賦予他只能在 dev namespace 中讀取（get, list）Pod 的權限。」
> - **Network Policy 題型**：「請建立一個 Network Policy，只允許帶有 `tier=frontend` 標籤的 Pod，連線到 `tier=backend` 的資料庫 Pod 的 3306 Port，其他流量一律阻擋。」

🛑 **避坑指南**：
> - **RBAC 綁定方向錯誤**：初學者極易搞混 RoleBinding 的綁定對象。記住：RoleBinding 是一個橋樑，它將「權限規則 (Role)」綁定給「人或機器 (User/ServiceAccount)」。
> - **憑證路徑迷宮**：在後續的 TLS 章節，考試環境的憑證路徑很複雜，千萬不要死背，善用 `cat /etc/kubernetes/manifests/kube-apiserver.yaml` 來找尋官方憑證的真實擺放位置。

🔧 **Troubleshooting (除錯方向)**：
> 如果發現 Pod 一直出現 `Forbidden` 或 `Unauthorized` 的錯誤日誌，第一時間請檢查該 Pod 使用的 ServiceAccount，並確認該帳號背後關聯的 Role 是否有對應的 API 操作權限 (verbs)。

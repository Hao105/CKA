# 180. Developing network policies (開發網路策略)

## 1. 🏷️ 課程定位
- **章節編號與名稱：** 第 7 節：Security (安全)
- **影片標題：** 180. Developing network policies (開發網路策略)

## 2. 📌 核心概念摘要
開發 Network Policy 就像是為 Kubernetes 內的 Pod 撰寫專屬的「分散式防火牆規則」。本節的核心目標在於實作：透過精準設定 `podSelector` (控制內部互動) 與 `ipBlock` (控制與外部系統的連線)，來規範入站 (Ingress) 與出站 (Egress) 流量，實現真正的微服務網路隔離。

## 4. 🔑 知識點擷取 (Detailed Notes)
根據截圖中的 YAML 架構，以下是您必須掌握的底層解析：

- **目標鎖定 (`spec.podSelector`)：**
  定義這張「防火牆規則」要套用在誰身上。截圖中的 `matchLabels: role: db` 代表此規則將套用於所有帶有該標籤的 Database Pod，這些 Pod 將進入隔離狀態。

- **策略類型 (`policyTypes`)：**
  必須明確宣告你要控制的方向（Ingress、Egress 或兩者皆寫）。
  ⚠️ **致命限制：** 如果你寫了 Egress 卻沒有給具體的 `egress` 規則陣列，預設行為是「拒絕所有出站流量 (Deny All)」。

- **入站流量 (`ingress.from`)：**
  控制「誰可以連線進來」。範例中使用 `podSelector`，允許帶有 `name: api-pod` 標籤的內部 Pod，透過 TCP 3306 埠存取 DB。

- **出站流量 (`egress.to`) 與外部系統：**
  控制「被隔離的 Pod 可以連去哪裡」。當目標不是叢集內的 Pod，而是外部機器（如截圖中的 Backup Server）時，必須使用 `ipBlock` 與 CIDR 格式 (例如 `192.168.5.10/32` 代表單一特定 IP)，並指定對應的 Port (80)。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
Network Policy 沒有辦法靠單一 `kubectl create` 指令完整生成。考場上請直接打開 Kubernetes 官方文件，搜尋 Network Policies 並複製 YAML 範本來修改。

不過，寫完 Policy 後的「驗證」是考試拿分的關鍵：

```bash
# 💡 驗證 Ingress (從其他 Pod 測試是否能連進 DB Pod)
# 開一個暫時的 busybox，使用 nc (netcat) 指令掃描指定 Port
kubectl run test-ingress --image=busybox:1.28 --rm -it -- nc -zv <db-pod-ip> 3306

# 💡 驗證 Egress (進入 DB Pod 內部，測試是否能連到外部 IP)
kubectl exec -it <db-pod-name> -- nc -zv 192.168.5.10 80

# 💡 檢查 Policy 狀態與選擇器是否正確綁定
kubectl describe networkpolicy db-policy
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
### 🎯 考試情境預測：
考題通常會給定一個 Namespace，要求你建立 Network Policy 來保護內部的 DB Pod。
條件可能包含：只允許來自 `namespace: frontend` 且標籤為 `app: web` 的 Pod 存取 3306 埠，並允許 DB Pod 連線到外部的特定 IP (`ipBlock`) 進行備份。

### 🛑 避坑指南 (YAML 陣列地獄)：
**AND 還是 OR？** 這絕對是考場最容易踩的坑：
- 當 `namespaceSelector` 和 `podSelector` 寫在 **同一個橫槓 `-` 下面** 時，代表 **AND (交集)**（必須同時符合這個 Namespace 且帶有這個 Label）。
- 若寫在 **不同橫槓 `-` 之下**，代表 **OR (聯集)**（符合這個 Namespace「或」帶有這個 Label 都可以）。

記得 `port` 屬性下的 `protocol` (如 TCP/UDP) 必須全大寫。

### 🔧 Troubleshooting：
- **如果套用了 Policy 但流量還是不通？**
  1. 執行 `kubectl describe netpol <name>`，觀察 PodSelector 欄位是否真的有抓到目標 Pod 的 IP（如果寫著 `<none>` 代表標籤打錯了）。
  2. 檢查被選中的目標 Pod，其對外的 Port 號是否與 Policy 中定義的相符。

## 2. 📌 核心概念摘要
Kubernetes 的 Security Context（安全上下文） 是用來定義 Pod 或 Container 執行時的作業系統層級權限與存取控制設定。它能確保容器以「最低權限」運行（例如指定非 root 使用者、限制特權模式等），是防範容器逃逸（Container Escape）並強化叢集底層資源隔離性的關鍵機制。

## 4. 🔑 知識點擷取 (Detailed Notes)
- **定義與作用邊界：** Security Context 決定了容器進程（Process）所擁有的作業系統層級屬性，包含 UID/GID、Linux Capabilities、AppArmor/SELinux profiles 等。

- **層級覆蓋機制 (Override)：**
  - **Pod 級別：** 套用於 Pod 內的所有容器。設定如 `runAsUser` 等會作為所有容器的預設值。
  - **Container 級別：** 僅套用於該單一容器。若與 Pod 級別設定衝突時，Container 級別的設定將覆蓋 (Override) Pod 級別的設定。

- **⚠️ 致命限制條件 (Limitations)：**
  - **Capabilities 限制：** 如同影片截圖中特別標示的 *Note: Capabilities are only supported at the container level and not at the POD level*。`capabilities` (如 `add: ["MAC_ADMIN"]` 或 `drop: ["ALL"]`) **絕對不能**設定在 Pod 層級。
  - **權限提升控制：** 透過設定 `allowPrivilegeEscalation: false` 可以確保子進程無法獲得比父進程更多的權限；而 `privileged: true` 則會讓容器擁有宿主機 (Node) 上的完整 root 權限，考試中請依照題目嚴格設定，切勿自行加入特權。

## 5. 💻 CKA 必備實作指令 (Imperative Commands)
在 CKA 考場中，Security Context 無法單純依靠一行指令完整建立，必須先生成 YAML 骨架再行編輯。

```bash
# 💡 CKA 考試技巧：先產生骨架 YAML，再手動加入 securityContext
kubectl run secure-pod --image=ubuntu --command --dry-run=client -o yaml -- sleep 3600 > pod.yaml

# 💡 快速檢視叢集中現有 Pod 的執行身分 (利用 custom-columns 快速除錯)
kubectl get pods -o custom-columns="NAME:.metadata.name,POD_USER:.spec.securityContext.runAsUser"
```

💡 考場上最快的做法是用 vim 修改剛剛產生的 `pod.yaml`，加入以下區塊：
```yaml
# spec:
#   securityContext:        <-- Pod 級別
#     runAsUser: 1000
#   containers:
#   - name: ubuntu
#     image: ubuntu
#     securityContext:      <-- Container 級別
#       capabilities:
#         add: ["MAC_ADMIN"]
```

## 6. 🚀 CKA 考試延伸與 Troubleshooting
### 🎯 考試情境預測：
- **情境 1：** 要求建立一個 Pod，並強制要求以特定的 User ID (例如 `runAsUser: 1010`) 執行，且需為特定容器新增 Linux Capability (如 `NET_ADMIN` 或 `SYS_TIME`)。
- **情境 2：** 給定一個已經存在但無法正常啟動的 Pod，要求修復它的權限問題（通常是 YAML 寫錯層級，或是權限不足導致無法寫入掛載的 Volume）。

### 🛑 避坑指南：
- **縮排地獄：** `spec.securityContext` 與 `spec.containers[].securityContext` 屬性名稱一模一樣。考場上最容易縮排錯誤，請務必再三確認你是對齊 `spec` 還是對齊 `- name`。
- **陣列格式遺漏：** `capabilities.add` 必須是一個 List (陣列) 格式 `["CAP_NAME"]`，少寫中括號會導致 API 解析失敗。

### 🔧 Troubleshooting：
- **若 Pod 狀態卡在 `CreateContainerConfigError`：**
  執行 `kubectl describe pod <pod-name>`，查看最下方的 Events 欄位。這通常表示你的 securityContext 寫錯了層級（例如把 capabilities 塞到 Pod 級別）。
- **若容器出現 `CrashLoopBackOff` 或 `Error`：**
  執行 `kubectl logs <pod-name>` 檢查日誌。通常是因為設定了 `runAsUser` 後，該 User ID 對於容器內的檔案系統或掛載的 Volume 沒有讀寫權限（此時可考慮配合考點 `fsGroup` 來解決）。

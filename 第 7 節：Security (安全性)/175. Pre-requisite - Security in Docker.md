# Pre-requisite - Security in Docker

## 📌 核心觀念

在 Docker 與 Kubernetes 中，預設的安全機制會把容器內的 root 權限「閹割」掉（移除大量 Linux Capabilities），讓它無法破壞宿主機 (Host Node)。
然而，加上 `--privileged`（在 K8s 中為 `privileged: true`）就像是**「拔除安全封印」**：這會完全解除容器的隔離限制，讓容器內的 root 使用者擁有與宿主機 root 完全相同的最高權限，雖然能帶來掛載硬體、修改網路等極大彈性，但也伴隨著最危險的安全風險。

*   **預設安全隔離**：預設情況下，Docker/Containerd 為了安全會大量閹割 Linux Capabilities，即使你是容器內的 root 也無法為所欲為。
*   **解除封印 (Privileged)**：加上特權後，容器將獲得「所有」的 Linux Capabilities，甚至可以存取宿主機上 `/dev` 目錄下的所有硬體設備。
*   **K8s 底層映射**：這個功能被封裝在 Pod YAML 的 `securityContext` 區塊中，設定為 `privileged: true`。
*   **必用場景**：通常用於網路外掛 (如 Calico, Flannel 修改路由)、儲存驅動 (CSI 掛載硬碟)，或是需要深度監控節點數據的底層 Agent。

## 💻 必考實戰指令

在 CKA 考場上，考題通常會要求建立一個擁有特權的 Pod。請善用 `--dry-run` 產生框架，再手動修改 YAML：

```bash
# 1️⃣ 考場快速產生 Pod YAML 框架
kubectl run privileged-pod --image=ubuntu --dry-run=client -o yaml > pod.yaml

# 2️⃣ 編輯 pod.yaml，在【容器層級】加入 securityContext (請參考下方 YAML 骨架)
vim pod.yaml

# 3️⃣ 套用建立
kubectl apply -f pod.yaml
```

## 🛡️ 實戰與最佳實踐 SOP

> [!IMPORTANT]
> **放錯層級陷阱 (致命細節)**：
> `securityContext` 在 Pod YAML 中有兩個地方可以放：
> 1. `spec.securityContext` (Pod 級別，套用於底下所有容器，例如設定共用的 `runAsUser`)
> 2. `spec.containers[].securityContext` (Container 級別)
> **⚠️ 考場死穴**：`privileged: true` **只能寫在 Container 級別的 `securityContext` 內**！ 如果你寫在 Pod 級別，`kubectl apply` 會直接報錯無法建立。無數考生在這裡浪費大量排錯時間。

> [!TIP]
> **Troubleshooting SOP：Apply 後出現 Forbidden？**
> 如果你 `apply` 之後 Pod 連 Pending 的狀態都沒有，直接跳出 `Error from server (Forbidden)`。
> **排查方向**：這代表該 Namespace 可能啟用了 **Pod Security Admission (PSA)** 的 Restricted 模式，強行封殺了特權容器。考試時若遇到，請先執行 `kubectl get ns <namespace-name> --show-labels` 檢查該 Namespace 的安全標籤設定，確認題目是否有其他的配套要求（例如要你改部署到別的 Namespace）。

## 📝 YAML 骨架

正確配置特權容器的標準寫法：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: privileged-pod
spec:
  containers:
  - name: ubuntu
    image: ubuntu
    command: ["sleep", "3600"]
    securityContext:            # ⚠️ 注意縮排位置：必須在 containers 清單內
      privileged: true          # ⚠️ 賦予等同 docker run --privileged 的權限
```

## 🧠 自我測驗

<details>
<summary><b>1. 考題要求你將 `privileged: true` 加到 YAML 裡面，請問你應該加在 `spec.securityContext` 還是 `spec.containers[].securityContext` 裡面？</b></summary>
解答：必須加在 `spec.containers[].securityContext`（Container 級別）。若不小心加在 Pod 級別會直接導致報錯。
</details>

<details>
<summary><b>2. 在預設（非 Privileged）的容器中，即使你是用 root 身分登入，為什麼還是無法修改宿主機的網卡或路由表？</b></summary>
解答：因為 Docker/Containerd 與 Kubernetes 預設會為了安全隔離，將容器的 `Linux Capabilities` 大量閹割（例如移除了 `CAP_NET_ADMIN` 權限）。
</details>

<details>
<summary><b>3. 考題要你建立一個特權 Pod，但 `kubectl apply` 時立刻跳出 `Forbidden` 錯誤，通常是因為叢集啟用了什麼安全防護機制？</b></summary>
解答：通常是因為部署目標的 Namespace 啟用了 `Pod Security Admission (PSA)` 或早期的 `PodSecurityPolicy (PSP)` 機制，進而進行了特權操作的阻擋。
</details>
